## 7. 表情系统设计

### 7.1 表情系统目标

表情系统需要让用户一眼感受到机器人是“活着的”。

第一版建议采用极简几何表情：

- 两只眼睛；
- 一张嘴；
- 可选眉毛；
- 背景呼吸动画；
- 简单粒子或光晕。

不建议第一版使用复杂数字人或 3D 模型。

### 7.2 表情组件

| 组件 | 说明 |
|---|---|
| LeftEye | 左眼 |
| RightEye | 右眼 |
| Mouth | 嘴巴 |
| Eyebrow | 可选眉毛 |
| Background | 背景氛围 |
| Particle | 可选情绪粒子 |
| MotionLayer | 屏幕整体轻微位移或弹跳 |

### 7.3 表情状态

| 表情 | 使用场景 |
|---|---|
| neutral | 默认待机 |
| happy | 开心、被夸奖 |
| listening | 倾听用户说话 |
| thinking | 思考中 |
| speaking | 回复中 |
| confused | 没听懂、识别失败 |
| caring | 用户疲惫、难过 |
| sleepy | 夜间、长时间待机 |
| dizzy | 被摇晃 |
| annoyed | 被频繁打扰或强烈摇晃 |
| charging | 充电中 |
| low_battery | 低电量 |
| sleeping | 倒扣、夜间休眠 |
| surprised | 突然被拿起或用户靠近 |
| focus | 专注陪伴模式 |

### 7.4 表情参数模型

```json
{
  "expression": "happy",
  "eye_shape": "crescent",
  "eye_scale": 1.1,
  "eye_distance": 1.0,
  "eye_direction": "center",
  "blink_speed": "normal",
  "mouth_shape": "smile",
  "mouth_open_level": 0.2,
  "background_pulse": true,
  "motion_effect": "bounce",
  "duration_ms": 2000
}
```

### 7.5 表情映射规则

| 情绪 | 表情 | 语音风格 | 震动 |
|---|---|---|---|
| happy | happy | lively | soft_tick |
| concerned | caring | gentle | soft_pulse |
| thinking | thinking | none | none |
| confused | confused | hesitant | none |
| sleepy | sleepy | slow | none |
| dizzy | dizzy | playful | dizzy_buzz |
| low_energy | low_battery | weak | none |
| charging | charging | relaxed | soft_tick |

## 7A. 重点章节：机器人通过显示器表达表情的实现技术细节与场景设计

### 7A.1 显示器表情的功能定义

本项目中的“显示器表情”，不是简单播放几张表情图片，而是让手机屏幕成为机器人的“脸”。

这张脸由多个可动画化元素组成：

```text
眼睛
嘴巴
眉毛，可选
脸部背景
光效
粒子
字幕
状态提示
```

它需要与语音、摄像头、触摸、传感器、情绪引擎联动。

完整链路如下：

```text
用户输入 / 摄像头事件 / 传感器事件 / 设备状态
    ↓
情绪引擎判断当前情绪
    ↓
选择表情状态
    ↓
表情状态映射为眼睛、嘴巴、眉毛、背景参数
    ↓
屏幕动画渲染
    ↓
与 TTS、震动、嘴型同步
```

例如，用户摇晃手机时：

```text
motion.shake
    ↓
emotion = dizzy
    ↓
expression = dizzy
    ↓
眼睛变成晕圈
嘴巴变成波浪形
屏幕轻微抖动
震动反馈
语音说：“哇，别晃啦。”
```

这才是“显示器表情系统”的完整意义。

---

### 7A.2 表情系统设计目标

#### 7A.2.1 让机器人看起来“活着”

即使用户没有说话，机器人也应该有轻微生命感，例如：

```text
慢速眨眼
眼睛轻微呼吸缩放
视线偶尔移动
背景轻微渐变
嘴巴偶尔微笑
```

这样可以让它不像一个静态 App，而像一个正在陪伴用户的小生命。

#### 7A.2.2 让用户理解机器人状态

用户应该通过表情快速判断机器人当前状态。

| 表情 | 用户理解 |
|---|---|
| 睁眼看着用户 | 我在 |
| 眼睛左右转 | 我在思考 |
| 半闭眼 | 我困了 / 夜间模式 |
| 晕圈眼 | 我被晃晕了 |
| 闭眼微笑 | 我在充电 / 很舒服 |
| 眼睛变大 | 我惊讶 / 我看见你了 |
| 嘴巴小幅开合 | 我正在说话 |
| 嘴巴闭合 | 我在听 / 安静 |

#### 7A.2.3 让情绪和交互更自然

同一句话配不同表情，用户感受完全不同。

例如机器人说：

> 我在呢。

不同表情带来的感受：

| 表情 | 感受 |
|---|---|
| 默认睁眼 | 正常回应 |
| 关心表情 | 温柔陪伴 |
| 困倦表情 | 夜间陪伴 |
| 开心表情 | 欢迎用户 |
| 晕圈表情 | 被摇晃后的俏皮反应 |

### 7A.3 技术实现总体架构

表情系统建议分为六层：

```text
Expression Event 输入层
    ↓
Expression State 表情状态层
    ↓
Expression Mapper 参数映射层
    ↓
Animation Controller 动画控制层
    ↓
Rendering Layer 渲染层
    ↓
Sync Layer 联动层
```

各层职责如下：

| 层级 | 职责 |
|---|---|
| 输入层 | 接收对话、视觉、语音、触摸、传感器事件 |
| 状态层 | 维护当前表情状态 |
| 映射层 | 将 emotion 转换为眼睛、嘴巴、背景参数 |
| 动画层 | 控制眨眼、嘴型、过渡、弹跳 |
| 渲染层 | 使用 Flutter / Rive / Lottie 绘制 |
| 联动层 | 与 TTS、震动、摄像头、传感器同步 |

### 7A.4 推荐技术路线

#### 7A.4.1 Flutter CustomPainter

MVP 最推荐使用 Flutter 的 `CustomPainter` 自绘眼睛、嘴巴和背景。

优点：

```text
性能好
可控性强
不用依赖大量素材
易于参数化
适合做极简表情
跨平台一致性较好
```

适合实现：

```text
眼睛形状变化
眼球方向变化
嘴巴开合
嘴巴弧线变化
背景呼吸动画
眨眼动画
屏幕轻微抖动
```

#### 7A.4.2 Rive 动画

Rive 适合做更精细的状态机动画。

优点：

```text
动画师可参与
状态机友好
适合做复杂表情过渡
运行时可调参数
```

适合后续版本：

```text
复杂眼神
可爱表情
状态机动画
品牌化角色
多套皮肤
```

#### 7A.4.3 Lottie 动画

Lottie 适合播放预制动画。

优点：

```text
接入简单
视觉效果好
适合一次性动效
```

适合：

```text
充电动画
低电量动画
惊讶动画
庆祝动画
睡眠动画
```

缺点是交互参数化能力不如 CustomPainter 和 Rive。

#### 7A.4.4 Unity / 3D Avatar

不建议 MVP 使用 Unity 或 3D Avatar。

原因：

```text
开发成本高
性能消耗大
和手机桌面常亮场景不匹配
过早复杂化
```

如果未来要做高级数字人版本，再考虑 Unity 或 3D 引擎。

### 7A.9 表情优先级机制

不同事件可能同时触发表情，例如机器人正在说话时手机低电量，或者用户摇晃手机时机器人正在回答。

因此需要设计表情优先级。

推荐优先级：

| 优先级 | 表情来源 | 示例 |
|---|---|---|
| P0 | 安全/系统状态 | 来电、权限错误、严重低电量 |
| P1 | 强传感器事件 | 摇晃、倒扣、拿起 |
| P2 | 语音播放状态 | speaking、interrupted |
| P3 | 用户主动对话情绪 | caring、happy、confused |
| P4 | 视觉事件 | 用户出现、靠近、离开 |
| P5 | 默认情绪状态 | neutral、sleepy、focus |
| P6 | 随机微表情 | 眨眼、视线游走 |

示例：

```text
正在 speaking
    +
用户强烈摇晃手机
    ↓
临时切换 dizzy
    ↓
说完“别晃啦”
    ↓
回到 neutral 或 interrupted
```

### 7A.10 动画过渡机制

表情切换不能突兀，应该有过渡。

#### 7A.10.1 基础过渡

```text
neutral → listening：眼睛睁大，嘴巴闭合
listening → thinking：眼睛左右移动
thinking → speaking：嘴巴开始开合
speaking → neutral：嘴巴闭合，回到微笑
```

#### 7A.10.2 过渡参数

```json
{
  "from": "neutral",
  "to": "happy",
  "transition": "smooth",
  "duration_ms": 300,
  "easing": "easeOut"
}
```

#### 7A.10.3 特殊过渡

| 过渡 | 表现 |
|---|---|
| neutral → surprised | 快速放大眼睛 |
| surprised → happy | 眼睛变月牙，嘴巴微笑 |
| neutral → sleepy | 逐渐半闭眼，降低亮度 |
| speaking → interrupted | 嘴巴立刻闭合，眼睛一怔 |
| neutral → dizzy | 屏幕震动，眼睛旋转 |
| low_battery → charging | 从委屈变成闭眼微笑 |

### 7A.11 自然生命感：微表情系统

即使没有用户输入，机器人也不能完全静止。

#### 7A.11.1 自动眨眼

```text
每 3-7 秒随机眨眼一次
困倦状态眨眼更慢
惊讶状态眨眼更少
开心状态眨眼更轻快
```

#### 7A.11.2 呼吸动画

```text
眼睛或背景轻微缩放
周期 2-4 秒
幅度很小
```

#### 7A.11.3 视线游走

```text
空闲时偶尔看左、看右
检测到用户时看向用户
思考时眼睛左右移动
```

#### 7A.11.4 情绪残留

表情不要立刻完全回到默认。

例如：

```text
用户夸奖机器人
    ↓
happy 维持 2 秒
    ↓
soft_happy 维持 5 秒
    ↓
neutral
```

这样机器人显得更自然。

### 7A.12 具体场景设计

#### 场景 1：默认待机

状态：

```text
用户没有说话
手机放在桌面
机器人处于 idle
```

表情：

```text
眼睛正常睁开
每隔几秒自然眨眼
嘴巴轻微微笑
背景轻微呼吸
```

价值：

让手机不是静态界面，而是有“在场感”。

---

#### 场景 2：用户唤醒

用户说：

> 小眨。

或双击屏幕。

表情变化：

```text
sleepy / neutral
    ↓
眼睛快速睁开
    ↓
看向用户
    ↓
嘴巴微笑
```

机器人说：

> 我在呢，你说。

---

#### 场景 3：正在倾听

用户长按屏幕或语音唤醒后开始说话。

表情：

```text
眼睛专注看向用户
嘴巴闭合
背景轻微脉冲
字幕显示“我在听”
```

如果用户停顿太久：

```text
眼睛轻轻眨一下
嘴巴仍闭合
不急着打断
```

---

#### 场景 4：正在思考

用户说完问题后，机器人等待 LLM 回复。

表情：

```text
眼睛左右移动
嘴巴平直
背景有轻微流动
可显示“想一想……”
```

如果等待时间超过 3 秒：

```text
眼睛稍微眯一下
显示“这个问题我需要想一下”
```

价值：

减少模型延迟带来的等待焦虑。

---

#### 场景 5：正在说话

TTS 开始播放。

表情：

```text
眼睛看向用户
嘴巴随语音开合
根据语气调整嘴巴幅度
字幕同步显示
```

如果是温柔语气：

```text
眼睛柔和
嘴巴开合幅度小
背景暖色
```

如果是开心语气：

```text
月牙眼
嘴巴上扬
整体轻微弹跳
```

---

#### 场景 6：用户靠近

视觉系统检测到用户靠近。

表情：

```text
眼睛变大
眼神聚焦
嘴巴微笑
```

机器人可以说：

> 你靠近啦，我看到你了。

或者不说话，只通过眼神回应。

---

#### 场景 7：用户离开

视觉系统检测到用户离开画面。

表情：

```text
眼睛左右看一下
嘴巴变成小平嘴
等待几秒
进入 sleepy 或 idle
```

如果用户长时间离开：

```text
眼睛半闭
背景变暗
进入省电状态
```

---

#### 场景 8：用户回来

用户重新出现在画面中。

表情：

```text
眼睛快速睁大
嘴巴微笑
背景轻微亮起
```

机器人说：

> 你回来啦。

如果之前有未完成任务，可以说：

> 要继续刚才的事吗？

---

#### 场景 9：被摇晃

加速度计检测到强烈摇晃。

表情：

```text
眼睛变成旋涡
嘴巴波浪形
屏幕轻微抖动
背景晃动
```

机器人说：

> 哇，别晃啦，我有点头晕。

说完后：

```text
dizzy 维持 2 秒
逐渐恢复 neutral
```

---

#### 场景 10：充电

检测到充电插入。

表情：

```text
闭眼微笑
背景呼吸发光
可显示能量条
```

机器人说：

> 补充能量中，我感觉好多啦。

---

#### 场景 11：低电量

电量低于阈值。

表情：

```text
眼睛半闭
嘴巴下弯
背景变暗
动画速度降低
```

机器人说：

> 我快没电了，能不能帮我充一下？

如果用户不处理，减少语音输出，只保留低电量提示。

---

#### 场景 12：夜间模式

时间较晚或光线很暗。

表情：

```text
眼睛半闭
背景低亮度
嘴巴动作变慢
字幕亮度降低
```

语音：

```text
音量降低
语速变慢
回复变短
```

价值：

避免夜间打扰。

---

#### 场景 13：用户夸奖机器人

用户说：

> 你真聪明。

表情：

```text
眼睛变成月牙
嘴巴上扬
整体轻微弹跳
背景轻微发光
```

机器人说：

> 嘿嘿，被你夸我有点开心。

注意不要过度拟人化或情感绑架。

---

#### 场景 14：识别失败

STT 或 LLM 没有理解用户。

表情：

```text
一只眼稍大
一只眼稍小
嘴巴变成小波浪
```

机器人说：

> 我刚刚没太听清，可以再说一遍吗？

价值：

用表情降低失败带来的挫败感。

---

#### 场景 15：用户要求安静

用户说：

> 别说话了。

表情：

```text
嘴巴闭合
眼睛轻轻眨一下
进入安静模式
```

机器人只显示字幕：

> 好，我安静陪着。

后续不再主动语音输出。

### 7A.15 性能与功耗设计

表情系统会长期显示，因此必须关注性能和耗电。

#### 7A.15.1 帧率策略

| 模式 | 帧率 |
|---|---|
| 正常交互 | 30-60 fps |
| 桌面待机 | 15-30 fps |
| 夜间模式 | 10-15 fps |
| 低电量模式 | 5-10 fps |
| 静态休眠 | 几乎停止动画 |

#### 7A.15.2 动画节流

不需要每一帧都重绘所有元素。

可以分层渲染：

```text
背景低频更新
眼睛中频更新
嘴巴说话时高频更新
字幕按文本更新
```

#### 7A.15.3 防 OLED 烧屏

如果长时间桌面常亮，需要防止 OLED 烧屏。

建议：

```text
整体脸部位置每隔几分钟轻微移动
背景亮度不要过高
避免长时间显示纯白高亮元素
夜间自动降低亮度
提供屏幕保护模式
```

---

#### 7A.15.4 微表情基线抖动机制

为了同时解决 OLED 烧屏和表情呆板问题，表情系统应引入“微表情基线抖动”。

核心思想：

- 眼睛、嘴巴、脸部主体不应长期停留在完全相同像素位置；
- 每隔一段时间通过低频正弦波、余弦波或 Perlin Noise 产生 2-5 像素级别的微小偏移；
- 该偏移应足够小，用户几乎感知不到，但能打破 OLED 像素长期固定点亮的问题；
- 该偏移也能让机器人看起来更像有轻微呼吸和生命感。

推荐公式：

```text
X_render = X_base + sin(ω1 × t) × A1
Y_render = Y_base + cos(ω2 × t) × A2
```

参数建议：

| 参数 | 建议值 | 说明 |
|---|---:|---|
| A1 | 2-5 px | 横向最大偏移 |
| A2 | 2-5 px | 纵向最大偏移 |
| ω1 | 0.03-0.08 | 横向低频速度 |
| ω2 | 0.02-0.06 | 纵向低频速度 |
| update_interval | 16-100 ms | 根据当前帧率策略调整 |

伪代码示例：

```dart
final dx = sin(timeSeconds * omegaX) * amplitudeX;
final dy = cos(timeSeconds * omegaY) * amplitudeY;

final renderCenter = Offset(
  baseCenter.dx + dx,
  baseCenter.dy + dy,
);
```

如果使用 Perlin Noise，可获得更自然、不完全周期化的漂移效果：

```text
X_render = X_base + noise(t × k1) × A1
Y_render = Y_base + noise(t × k2 + offset) × A2
```

MVP 推荐：

```text
优先使用正弦 / 余弦低频偏移
振幅控制在 2-3 px
夜间和低电量模式下进一步降低更新频率
```

验收标准：

| 验收项 | 标准 |
|---|---|
| 微位移可用 | 长时间待机时眼睛主体像素位置会缓慢变化 |
| 用户无明显感知 | 不应产生明显晃动或眩晕感 |
| 防烧屏 | 常亮模式下避免核心高亮元素长期固定 |
| 生命感 | 待机时表情略有呼吸感，而非完全静止 |


### 7A.16 MVP 推荐实现范围

第一版表情系统建议实现：

1. 两只眼睛；
2. 一张嘴；
3. 自动眨眼；
4. 默认、倾听、思考、说话、开心、关心、困惑、困倦、晕眩、充电、低电量、睡眠表情；
5. 嘴巴说话开合；
6. 眼睛跟随用户位置；
7. 摇晃晕眩表情；
8. 充电和低电量表情；
9. 夜间低亮度表情；
10. 字幕显示；
11. 表情调试面板。

暂不建议 MVP 做：

```text
复杂 3D 数字人
高精度面部骨骼动画
复杂皮肤系统
音素级唇形
大量粒子特效
多人表情交互
```

### 7A.17 表情功能验收标准

| 验收项 | 标准 |
|---|---|
| 默认生命感 | 待机时有自然眨眼和轻微呼吸 |
| 表情切换 | 不同状态能切换到对应表情 |
| 语音同步 | 说话时嘴巴开合，结束后闭合 |
| 视觉联动 | 用户左右移动时眼睛方向变化 |
| 触摸反馈 | 点击眼睛或嘴巴有反应 |
| 传感器联动 | 摇晃手机进入 dizzy 表情 |
| 设备联动 | 充电和低电量有对应表情 |
| 夜间模式 | 表情亮度降低，动画变慢 |
| 动画性能 | 正常状态保持 30fps 左右 |
| 功耗控制 | 桌面模式不会明显发热 |
| 防打扰 | 安静模式下嘴巴闭合，不主动说话 |
| 调试能力 | 开发者可手动切换表情 |

### 7A.18 小结

机器人通过显示器表达表情的核心，是把屏幕从“信息显示器”变成“机器人的脸”。

最小可行路径是：

```text
Emotion / Event
    ↓
ExpressionState
    ↓
眼睛参数 + 嘴巴参数 + 背景参数
    ↓
Flutter / Rive / Lottie 渲染
    ↓
与语音、视觉、触摸、传感器同步
```

MVP 阶段最重要的是做好：

```text
眼睛会看
嘴巴会说
表情会变
状态有连续性
动画不僵硬
长时间显示不耗电
```

只要实现这些，手机就会从一个普通聊天界面，变成一个真正有“脸”、有“状态”、有“情绪”的 AI 伙伴。
