## 8. 语音系统设计

### 8.1 语音输入流程

```text
用户说话
  ↓
唤醒词 / 点击麦克风 / 长按屏幕
  ↓
录音
  ↓
STT 语音识别
  ↓
文本发送至 Dialogue Manager
  ↓
LLM 生成结构化响应
```

### 8.2 语音输出流程

```text
AI 返回 reply_text + voice_style
  ↓
TTS 生成语音
  ↓
播放语音
  ↓
音量驱动嘴巴开合
  ↓
播放完成后回到 idle
```

### 8.3 语音状态

| 状态 | 说明 |
|---|---|
| idle | 空闲 |
| listening | 正在听 |
| recognizing | 正在识别 |
| thinking | 等待模型 |
| speaking | 正在说话 |
| interrupted | 被用户打断 |
| error | 语音错误 |

### 8.4 语音风格参数

```json
{
  "voice_style": "gentle",
  "speed": 0.9,
  "pitch": 1.0,
  "volume": 0.8
}
```

### 8.5 打断机制

用户在机器人说话时可以：

- 点击屏幕打断；
- 说“停一下”打断；
- 长按进入重新倾听；
- 倒扣手机进入静音或睡眠。

打断后状态变更：

```text
speaking → interrupted → listening / idle
```

## 8A. 重点章节：机器人“说”的功能实现技术细节与场景设计

### 8A.1 “说”的功能定义

本项目中的机器人“说”，不是简单把文字通过 TTS 念出来，而是一个完整的多模态表达系统。

完整链路如下：

```text
AI 生成回复内容
    ↓
判断说话情绪和语气
    ↓
选择语音音色、语速、音高、停顿
    ↓
TTS 生成语音
    ↓
手机扬声器播放
    ↓
屏幕嘴巴动画同步开合
    ↓
眼睛、表情、背景和震动共同配合
```

因此，“说”的核心不是单一 TTS 能力，而是：

```text
文本内容 + 语音情绪 + 嘴型动画 + 表情状态 + 场景语气 + 打断控制
```

例如，同一句“我在呢”，在不同场景中应该有不同的表达方式：

| 场景 | 说法 | 表情 |
|---|---|---|
| 用户普通唤醒 | 我在呢，你说。 | 默认睁眼 |
| 用户疲惫 | 我在呢，慢慢说。 | 温柔关心 |
| 用户半夜说话 | 嗯……我在，小声一点陪你。 | 困倦低亮度 |
| 用户摇晃手机 | 哇，别晃啦，我有点头晕。 | 晕圈眼 |
| 充电中 | 正在补充能量，我感觉好多啦。 | 闭眼微笑 |

---

### 8A.2 技术实现总体流程

机器人“说”的技术链路建议分为七层：

```text
Dialogue Response 对话生成
    ↓
Speech Policy 说话策略
    ↓
Voice Style Mapping 语音风格映射
    ↓
TTS Synthesis 语音合成
    ↓
Audio Playback 音频播放
    ↓
Mouth Animation 嘴巴动画
    ↓
Expression & Haptic Sync 表情和震动同步
```

各层职责如下：

| 层级 | 职责 |
|---|---|
| 对话生成层 | LLM 生成回复内容 |
| 说话策略层 | 判断是否应该说、说多长、用什么语气 |
| 语音风格层 | 映射音色、语速、音高、音量、情绪 |
| TTS 层 | 将文本转成语音 |
| 播放层 | 控制扬声器播放、暂停、停止、打断 |
| 嘴型层 | 根据播放状态、音频能量或音素驱动嘴巴动画 |
| 联动层 | 表情、眼神、背景、震动同步变化 |

---

### 8A.3 LLM 回复生成

机器人说话内容来自对话大脑。LLM 不应只返回文本，而应返回结构化结果，让客户端知道：

- 要说什么；
- 是否要说出声；
- 用什么语气说；
- 用什么表情配合；
- 嘴巴如何动；
- 是否震动；
- 是否允许用户打断。

推荐输出格式：

```json
{
  "text": "听起来你今天消耗挺大。要不要我陪你简单复盘一下？",
  "emotion": "concerned",
  "expression": "caring",
  "voice": {
    "style": "gentle",
    "speed": 0.9,
    "pitch": 0.95,
    "volume": 0.8
  },
  "mouth_action": "talking_soft",
  "haptic": "soft_pulse",
  "should_speak": true,
  "allow_interrupt": true
}
```

这样 App 不需要猜测语气，而是直接按照结构化参数执行。

### 8A.4 说话策略 Speech Policy

不是所有 AI 回复都应该被朗读出来。机器人需要根据当前场景判断是否适合发声。

判断因素包括：

- 当前是否处于夜间模式；
- 用户是否开启静音；
- 用户是否处于专注模式；
- 当前是否在公共场所模式；
- 用户是否主动语音提问；
- 回复内容是否过长；
- 设备电量是否过低；
- 用户是否倒扣手机；
- 用户是否说过“安静一点”；
- 当前是否正在通话或播放其他系统音频。

#### 8A.4.1 典型说话策略

| 场景 | 说话策略 |
|---|---|
| 用户主动语音提问 | 默认语音回复 |
| 用户文字输入 | 可只显示字幕，或按设置朗读 |
| 夜间模式 | 降低音量，缩短回复 |
| 专注模式 | 少说话，尽量简短 |
| 低电量 | 减少长语音 |
| 用户倒扣手机 | 不说话，进入睡眠 |
| 用户正在通话 | 不播放语音 |
| 公共场所模式 | 默认字幕，不外放 |
| 用户靠近手机 | 可小声说话 |
| 用户说“安静一点” | 进入低音量或静音模式 |

#### 8A.4.2 策略输出示例

```json
{
  "should_speak": true,
  "speech_mode": "whisper",
  "max_sentence_count": 2,
  "show_subtitle": true,
  "allow_interrupt": true
}
```

### 8A.5 语音风格映射 Voice Style Mapping

机器人说话时，语音风格应与情绪状态一致。

| emotion | voice.style | speed | pitch | volume |
|---|---|---:|---:|---:|
| neutral | neutral | 1.0 | 1.0 | 0.8 |
| happy | lively | 1.05 | 1.05 | 0.9 |
| concerned | gentle | 0.9 | 0.95 | 0.75 |
| sleepy | slow | 0.8 | 0.9 | 0.55 |
| surprised | lively | 1.1 | 1.1 | 0.9 |
| dizzy | playful | 0.95 | 1.05 | 0.8 |
| annoyed | flat | 0.95 | 0.95 | 0.75 |
| charging | relaxed | 0.9 | 1.0 | 0.7 |
| low_battery | weak | 0.8 | 0.9 | 0.55 |
| focus | calm | 0.9 | 0.95 | 0.6 |

MVP 阶段可以不追求真正的情绪化 TTS，而是先通过以下方式营造情绪：

- 语速；
- 音量；
- 音高；
- 停顿；
- 回复文案；
- 嘴巴动画；
- 眼神和表情；
- 背景氛围；
- 震动反馈。

### 8A.6 TTS 技术选择

#### 8A.6.1 手机系统 TTS

MVP 最推荐先用手机系统 TTS。

```text

```

- 情绪表现有限；
- 音色可控性有限；
- 中文自然度取决于系统；
- 跨设备表现不一致。

适用阶段：


优点：

### 8A.8 嘴巴动画同步

“说”的体验不能只有声音，屏幕上的嘴巴必须跟随语音动起来。

#### 8A.8.1 MVP：播放状态驱动嘴型

最简单方案是根据 TTS 播放状态驱动嘴巴开合。

流程：

```text
TTS 播放开始
    ↓
进入 speaking 表情
    ↓
嘴巴按模拟节奏开合
    ↓
TTS 播放结束
    ↓
嘴巴闭合，回到 idle
```

嘴巴参数示例：

```json
{
  "mouth_shape": "talking",
  "open_level": 0.6,
  "speed": "normal"
}
```

MVP 阶段可以先不读取真实音频波形，而是用模拟节奏：

```text
短句：嘴巴快速开合 1-2 秒
长句：嘴巴持续开合
停顿：嘴巴微闭
结束：嘴巴闭合微笑
```

优点是实现简单，适合第一版。

#### 8A.8.2 进阶：音频能量驱动

如果可以获取 TTS 音频数据，可分析音频振幅，驱动嘴巴开合。

```text
Audio PCM / waveform
    ↓
计算 RMS 音量
    ↓
映射 mouth_open_level
    ↓
每帧更新嘴巴开合
```

映射示例：

```text
RMS 低 → mouth_open_level = 0.2
RMS 中 → mouth_open_level = 0.5
RMS 高 → mouth_open_level = 0.9
```

这种方式比模拟节奏更自然。

#### 8A.8.2A MVP 推荐：音频振幅平滑法

虽然模拟节奏实现简单，但第一版建议直接采用“音频能量驱动 + 平滑滤波”的嘴型同步方案。原因是模拟节奏容易出现明显“对不上口型”的机械感，用户会觉得机器人在假动嘴。

推荐流程：

```text
TTS 音频播放
    ↓
获取当前播放帧 PCM 数据
    ↓
计算 RMS 均方根振幅
    ↓
归一化为 0-1 的 RMS_normalized
    ↓
一阶低通滤波平滑
    ↓
映射为 mouth_open_level
    ↓
驱动嘴巴开合
```

RMS 计算：

```text
RMS = sqrt((x1² + x2² + ... + xn²) / n)
```

为了防止嘴巴过度闪烁，需要对开合值做一阶低通滤波：

```text
level_new = α × level_old + (1 - α) × RMS_normalized
```

推荐参数：

| 参数 | 建议值 | 说明 |
|---|---:|---|
| α | 0.7 | 平滑系数，越大越稳定 |
| RMS window | 20-50 ms | 每次计算的音频窗口 |
| min_open | 0.05 | 静音时嘴巴最小开合 |
| max_open | 0.85 | 最大张嘴程度 |
| update_rate | 30 fps | 嘴型刷新频率 |

伪代码示例：

```dart
double smoothMouthLevel({
  required double oldLevel,
  required double rmsNormalized,
  double alpha = 0.7,
}) {
  final next = alpha * oldLevel + (1 - alpha) * rmsNormalized;
  return next.clamp(0.05, 0.85);
}
```

Flutter 实现建议：

| 方案 | 说明 |
|---|---|
| flutter_sound | 用于音频播放和 PCM 数据处理，适合自管音频流 |
| 原生平台插件 | iOS/Android 分别获取播放音频能量，效果更稳 |
| 后端 TTS 返回 mouth_timeline | 后端直接生成嘴型时间线，前端按时间播放 |
| 系统 TTS fallback | 无法获取 PCM 时退回模拟嘴型 |

MVP 推荐决策：

```text
优先使用 RMS 音频能量驱动嘴巴
如果系统 TTS 无法暴露 PCM，则使用“播放状态 + 文本长度 + 标点停顿”的模拟嘴型 fallback
```

验收标准：

| 验收项 | 标准 |
|---|---|
| 嘴巴随音量变化 | 声音大时张嘴更大，停顿时嘴巴收小 |
| 无高频抖动 | 嘴巴不会疯狂闪烁 |
| 停顿自然 | 标点或静音段嘴巴接近闭合 |
| fallback 可用 | 无法获取 PCM 时仍能用模拟嘴型说话 |


#### 8A.8.3 高级：音素级口型同步

高级版本可基于音素或 viseme 实现更准确的口型同步。

```text
文本
    ↓
TTS / phoneme aligner
    ↓
音素序列
    ↓
viseme 口型序列
    ↓
嘴巴动画
```

音素和嘴型示例：

| 音素 | 嘴型 |
|---|---|
| a | 张大嘴 |
| o | 圆嘴 |
| i | 横向嘴 |
| m/b/p | 闭嘴 |
| f/v | 咬唇 |
| silence | 闭嘴 |

第一版不建议做音素级口型同步，除非产品已经验证成立并进入高级表现阶段。

### 8A.9 语音、表情和震动联动

机器人说话时，眼睛、嘴巴、背景和触觉反馈应同步变化。

| 语音状态 | 表情表现 |
|---|---|
| 准备说话 | 眼睛看向用户，嘴巴微张 |
| 正在说话 | 嘴巴开合，眼睛偶尔眨眼 |
| 温柔说话 | 眼睛变软，嘴巴小幅开合 |
| 开心说话 | 眼睛弯起，嘴巴上扬 |
| 思考后说话 | 眼睛先左右移动，再开口 |
| 被打断 | 嘴巴立刻闭合，眼睛一怔 |
| 低声说话 | 眼睛半闭，嘴巴小幅开合 |
| 生气/不满 | 眼睛变窄，嘴巴较平 |
| 困倦说话 | 慢眨眼，嘴巴动作慢 |

#### 8A.9.1 说话时的 Haptic 反馈

震动不能过度使用，只建议用于关键情绪或设备事件：

| 场景 | 震动 |
|---|---|
| 开心确认 | soft_tick |
| 温柔安慰 | soft_pulse |
| 强烈摇晃回应 | dizzy_buzz |
| 低电量提醒 | alert_tick |
| 普通说话 | none |

### 8A.10 具体场景设计

#### 场景 1：用户主动聊天

用户说：

> 我今天有点累。

系统流程：

```text
STT 识别文本
    ↓

    ↓
生成关心回复
    ↓
TTS 用 gentle 风格播放
    ↓
嘴巴轻柔开合
    ↓
眼睛呈关心状态
```

机器人回复：

> 听起来你今天消耗挺大。要不要我陪你简单复盘一下？

表现：

- 眼睛微微下垂；
- 嘴巴小幅开合；
- 语速稍慢；
- 音量稍低；
- 轻微震动一次。

#### 场景 6：夜间低声陪伴

夜间模式下，用户轻声说：

> 睡不着。

机器人说：

> 嗯，我在。要不要我小声陪你聊一会儿？

表现：

- 屏幕亮度降低；
- 眼睛半闭；
- 音量降低；
- 语速放慢；
- 回复变短；
- 不使用活泼语气。

---

#### 场景 7：专注模式提醒

用户工作 45 分钟后，机器人轻声说：

> 你已经专注一会儿了，要不要喝口水？我只提醒这一次。

表现：

- 表情克制；
- 音量较低；
- 语气平静；
- 不连续追问；
- 如果用户无回应，自动安静。

---

#### 场景 8：用户触摸嘴巴区域

用户点击或遮住屏幕嘴巴区域。

机器人说：

> 唔……别捂我嘴呀。

表现：

- 嘴巴变小；
- 眼睛眨一下；
- 语气俏皮；
- 短震反馈。

如果用户频繁触摸嘴巴区域，则改为：

> 好啦好啦，我先安静一点。

并进入低打扰状态。

---

#### 场景 9：用户要求安静

用户说：

> 你别说话，用文字就行。

机器人回复一次：

> 好，我改成安静模式。

之后：

- 不再语音播放；
- 只显示字幕；
- 嘴巴不动或轻微动；
- 表情仍保留。

---

#### 场景 10：用户举起物品问问题

用户说：

> 你看看这是什么？

视觉模型识别后，机器人说：

> 我看到你拿着一个杯子，看起来像是不锈钢保温杯。

表现：

- 先进入认真看的表情；
- 短暂停顿；
- 再开口说明；
- 如果不确定，说“我不太确定”。

### 8A.11 主动说话设计

机器人可以主动说话，但必须克制。

#### 8A.11.1 主动说话触发条件

| 触发 | 示例 |
|---|---|
| 用户回来 | 你回来啦。 |
| 长时间专注 | 要不要休息一下？ |
| 低电量 | 我快没电了。 |
| 充电开始 | 补充能量中。 |
| 光线太暗 | 要不要开个灯？ |
| 用户情绪低落 | 感觉你可能有点累。 |
| 定时提醒 | 该喝水了。 |

#### 8A.11.2 主动说话频率限制

建议加入频率控制：

```json
{
  "proactive_speech_cooldown_seconds": 300,
  "max_proactive_speech_per_hour": 3,
  "night_mode_proactive_speech": false
}
```

#### 8A.11.3 主动说话优先级

| 优先级 | 类型 |
|---|---|
| P0 | 安全提醒、低电量、用户明确设置提醒 |
| P1 | 用户回来、长时间专注、光线过暗 |
| P2 | 情绪陪伴、轻度问候 |
| P3 | 数字宠物式闲聊 |

P2 和 P3 必须严格限制频率。

### 8A.15A 语音打断的 AEC 与 VAD 工程要求

文档中虽然定义了“用户说停一下即可打断机器人说话”，但实际实现中存在一个关键工程瓶颈：

当手机扬声器正在播放 TTS 时，麦克风会采集到机器人自己的声音。如果没有回声消除和本地语音激活检测，系统很可能把机器人自己的 TTS 当成用户语音，或者完全听不到用户真正说出的“停一下”。

因此，语音打断必须加入 AEC、VAD 和关键词检测的组合策略。

#### 8A.15A.1 核心问题

```text
机器人 TTS 外放
    ↓
扬声器声音进入麦克风
    ↓
麦克风采集到“机器人自己的声音 + 用户声音 + 环境噪声”
    ↓
如果没有 AEC/VAD
    ↓
打断识别失败或误触发
```

典型失败情况：

| 情况 | 结果 |
|---|---|
| TTS 音量较大 | 用户说“停一下”被机器人声音淹没 |
| 无回声消除 | 系统误以为机器人自己的声音是用户输入 |
| 无本地 VAD | 一直上传噪声给 STT，功耗和延迟升高 |
| 云端 STT 延迟过高 | 用户打断不及时，体验失效 |

#### 8A.15A.2 推荐打断链路

```text
TTS speaking
    ↓
开启本地低功耗 VAD
    ↓
AEC 消除扬声器回声
    ↓
检测到用户语音活动
    ↓
本地关键词检测：停一下 / 别说了 / 安静 / pause / stop
    ↓
立即 stop TTS
    ↓
表情切换 interrupted
    ↓
进入 listening 或 quiet mode
```

#### 8A.15A.3 AEC 回声消除

AEC 的目标是从麦克风输入中尽量减去扬声器正在播放的 TTS 音轨。

实现建议：

| 平台 | 实现方向 |
|---|---|
| iOS | 使用 AVAudioSession 的 voiceChat / playAndRecord 模式 |
| Android | 使用 AcousticEchoCanceler、NoiseSuppressor、AutomaticGainControl |
| Flutter | 通过原生插件封装平台 AEC 能力 |
| WebRTC | 使用 WebRTC Audio Processing Module 做 AEC/NS/AGC |

推荐策略：

```text
MVP 阶段至少启用系统级 AEC
正式版考虑引入 WebRTC APM 作为跨平台音频处理层
```

#### 8A.15A.4 VAD 语音激活检测

VAD 用于判断当前麦克风输入中是否有“真人说话”。

要求：

- VAD 必须本地运行；
- VAD 在 TTS 播放期间保持低功耗监听；
- 只有检测到疑似用户语音时，才进入关键词识别或 STT；
- 不能把 TTS 外放本身当作用户说话。

推荐实现：

| 方案 | 说明 |
|---|---|
| WebRTC VAD | 经典轻量方案 |
| Silero VAD | 效果好，但端侧成本略高 |
| 平台原生语音检测 | 接入简单，但能力差异较大 |
| 关键词小模型 | 专门识别“停一下”等打断词 |

#### 8A.15A.5 打断关键词

MVP 建议内置本地打断词：

```text
停
停一下
别说了
安静
闭嘴
暂停
不用说了
stop
pause
quiet
```

用户说出打断词后，不需要等待完整云端 LLM 回复，应立即执行：

```json
{
  "event_type": "speech.interrupted",
  "reason": "local_keyword",
  "action": "stop_tts",
  "next_expression": "interrupted"
}
```

#### 8A.15A.6 降级策略

如果当前设备无法可靠支持 AEC/VAD，则使用备用交互：

| 条件 | 降级方式 |
|---|---|
| AEC 不可用 | 显示明显的“点按可打断”按钮 |
| VAD 不稳定 | 仅支持触摸打断 |
| TTS 外放过大 | 自动降低播放音量 |
| 耳机连接 | 优先支持语音打断，因为回声更少 |
| 公共场所模式 | 默认字幕，不外放 |

MVP 验收标准：

| 验收项 | 标准 |
|---|---|
| 触摸打断 | 点击或长按可立即停止 TTS |
| 关键词打断 | 中等音量下可识别“停一下” |
| 回声抑制 | 机器人自己的 TTS 不应频繁误触发 |
| 降级提示 | 语音打断不可用时显示触摸打断提示 |
| 延迟 | 本地打断响应应小于 300ms |


### 8A.15 MVP 推荐实现范围

第一版“说”的功能建议只做以下内容：

1. 使用系统 TTS；
2. 支持中文语音播放；
3. 支持语速、音量、音高基础控制；
4. 支持播放开始、播放结束回调；
5. 支持用户点击或长按打断；
6. 支持嘴巴在说话时开合；
7. 支持不同情绪映射不同说话风格；
8. 支持夜间降低音量；
9. 支持静音模式；
10. 支持字幕同步显示。

MVP 不建议做：

- 复杂情绪化 TTS；
- 音素级口型同步；
- 语音克隆；
- 多角色实时切换；
- 长文本朗读；
- 后台持续播报。

---

### 8A.16 语音功能验收标准

| 验收项 | 标准 |
|---|---|
| 语音播放 | 可以朗读 AI 回复 |
| 语音风格 | happy/gentle/sleepy 等至少有语速、音量差异 |
| 嘴巴动画 | 播放时嘴巴开合，结束后闭合 |
| 打断 | 点击或长按可停止语音 |
| 夜间模式 | 音量降低，回复变短 |
| 静音模式 | 不播放语音，只显示字幕 |
| 错误处理 | TTS 失败时显示文字 |
| 状态同步 | speaking 时表情和嘴巴一致 |
| 主动说话 | 有冷却时间，不能频繁打扰 |
| 隐私体验 | 用户可关闭语音播报 |

### 8A.17 小结

机器人“说”的实现，本质上是把 AI 回复转化成一个完整的多模态表达动作。

最小可行链路是：

```text
LLM 生成回复
    ↓
返回 text + voice_style + expression
    ↓
手机端 TTS 播放
    ↓
嘴巴动画同步
    ↓
表情和震动配合
```

MVP 阶段不需要一开始追求高级语音克隆或音素级口型同步。第一版只要实现：

```text
能自然朗读
能根据情绪调整语速和音量
说话时嘴巴会动
用户可以打断
夜间和静音模式可控
```

机器人就已经具备“会说话”的基础生命感。
