### 9.3 动作识别规则

## 10A. 重点章节：机器人“看”的功能实现原理与场景设计

### 10A.1 “看”的功能定义

本项目中的“看”不是简单打开摄像头显示画面，而是让机器人通过手机摄像头获得对用户、环境和物体的视觉理解，并将视觉结果转化为可驱动表情、语音、对话和行为的结构化事件。

完整链路如下：

```text
摄像头采集画面
    ↓
视觉算法识别画面内容
    ↓
转化为结构化视觉状态与视觉事件
    ↓
输入情绪引擎和对话系统
    ↓
驱动机器人眼神、表情、语音、震动和主动行为
```

例如，用户走到手机前时，不需要说话，机器人通过前置摄像头检测到“有人靠近”，然后屏幕眼睛睁大、视线转向用户，并用语音说：

> 你回来啦。

这类体验是本产品从普通聊天 App 升级为“手机具身情感机器人”的关键。

---

### 10A.2 技术实现总体流程

机器人视觉能力建议分为五层：

```text
Camera Capture 摄像头采集层
    ↓
Vision Detection 视觉识别层
    ↓
Vision State Mapping 视觉状态映射层
    ↓
Vision Event Bus 视觉事件分发层
    ↓
Expression / Dialogue / Emotion 联动层
```

各层职责如下：

| 层级 | 职责 | 输出 |
|---|---|---|
| 摄像头采集层 | 获取前置摄像头图像帧 | camera frame |
| 视觉识别层 | 检测人脸、位置、光线、物体等 | raw detection result |
| 视觉状态映射层 | 将识别结果转为业务状态 | VisionState |
| 视觉事件分发层 | 判断状态变化并发布事件 | VisionEvent |
| 联动层 | 驱动表情、对话、情绪、语音 | expression / reply / action |

### 10A.3 摄像头采集层

手机前置摄像头是机器人的“眼睛”。App 需要通过手机摄像头周期性获取图像帧，用于视觉识别。

Flutter 中可使用：

```text
camera
google_mlkit_face_detection
permission_handler
```

也可以在 iOS 和 Android 端分别使用平台原生能力：

```text
iOS：Vision Framework
Android：ML Kit Face Detection
跨平台：MediaPipe / Google ML Kit Flutter 插件
```

基础流程：

```text
申请摄像头权限
    ↓
打开前置摄像头
    ↓
获取实时图像帧
    ↓
抽帧或降分辨率
    ↓
送入视觉识别模块
```

为了控制功耗和发热，不建议持续高频分析高清视频。

推荐策略：

| 模式 | 视觉检测策略 |
|---|---|
| 普通聊天模式 | 用户唤醒或说话时短时启用 |
| 桌面陪伴模式 | 每 1-3 秒低频检测一次 |
| 专注模式 | 低频检测，只判断用户是否在场 |
| 夜间模式 | 默认关闭或极低频检测 |
| 隐私模式 | 完全关闭摄像头 |
| 用户明确请求“你看看” | 抓取单帧图像进行视觉理解 |

MVP 阶段推荐使用低分辨率图像，例如：

```text
320 × 240
640 × 480
```

这些分辨率已经足够做人脸存在、位置、距离和光线判断。

### 10A.3A 音视频并发下的功耗与发热动态降级策略

将手机作为“桌面常亮、常听、可视觉感知”的设备时，核心瓶颈不是单一算法精度，而是音视频流并发带来的发热、功耗和降频问题。

高风险组合：

```text
前置摄像头持续采集
    +
人脸检测持续运行
    +
麦克风持续唤醒监听 / 降噪 / VAD
    +
屏幕常亮动画渲染
    +
网络请求 / LLM 等待
    ↓
15-30 分钟内可能明显发热、掉帧、降频、耗电
```

因此，视觉系统必须具备动态降级能力，不能只依赖低分辨率。

#### 10A.3A.1 视觉检测动态频率

视觉检测频率应根据用户是否在场、最近交互时间、电量、温度和模式动态调整。

推荐状态机：

| 状态 | 条件 | 检测频率 |
|---|---|---:|
| active_tracking | 用户在画面中，正在互动 | 0.5-1 秒/次 |
| desktop_idle | 用户在画面中，但无交互 | 2-3 秒/次 |
| no_face_recent | 最近 30 秒未检测到人脸 | 5 秒/次 |
| no_face_long | 超过 3 分钟未检测到人脸 | 10-15 秒/次 |
| thermal_save | 设备发热或低电量 | 15-30 秒/次或关闭 |
| privacy/off | 隐私模式或用户关闭 | 不检测 |

推荐滑落逻辑：

```text
检测到人脸
    ↓
active_tracking / desktop_idle
    ↓
连续 30 秒无人脸
    ↓
no_face_recent：每 5 秒检测
    ↓
连续 3 分钟无人脸
    ↓
no_face_long：每 10-15 秒检测
    ↓
检测到人脸
    ↓
立即恢复 active_tracking
```

#### 10A.3A.2 多模态资源调度原则

摄像头、麦克风、屏幕动画不能同时长期满功率运行。

推荐策略：

| 资源 | 降级方式 |
|---|---|
| 摄像头 | 抽帧、低分辨率、无人时降频、发热时关闭 |
| 人脸检测 | 从连续检测改为定时采样 |
| 麦克风 | 本地 VAD 低功耗监听，避免持续云端 STT |
| 屏幕动画 | 待机降到 15-30 fps，低电量降到 5-10 fps |
| 网络 | 只在用户明确输入或重要事件时请求 Gateway |
| LLM | 长时间空闲时不保活重模型连接 |

#### 10A.3A.3 温度与电量触发降级

App 应监听设备电量、充电状态、低电量模式和系统热状态。

推荐规则：

| 条件 | 动作 |
|---|---|
| 电量 < 20% 且未充电 | 降低摄像头频率、降低动画帧率 |
| 电量 < 10% | 关闭视觉检测，仅保留触摸唤醒 |
| 系统进入低电量模式 | 禁用主动视觉，降低屏幕动画 |
| 设备发热 | 降低视觉检测到 15-30 秒/次 |
| 充电且屏幕常亮 | 控制亮度，避免边充边高负载 |

#### 10A.3A.4 端侧调度伪代码

```dart
Duration getVisionInterval(DeviceRuntimeState state) {
  if (!state.visionEnabled || state.privacyMode) {
    return Duration.zero;
  }

  if (state.isThermalHot || state.batteryLevel < 10) {
    return const Duration(seconds: 30);
  }

  if (!state.userPresentForLongTime) {
    return const Duration(seconds: 10);
  }

  if (!state.userPresentRecently) {
    return const Duration(seconds: 5);
  }

  if (state.isUserInteracting) {
    return const Duration(milliseconds: 800);
  }

  return const Duration(seconds: 3);
}
```

#### 10A.3A.5 MVP 验收标准

| 验收项 | 标准 |
|---|---|
| 无人时降频 | 连续无人 30 秒后，检测频率自动降低到 5 秒级 |
| 长时间无人 | 超过 3 分钟无人后，检测频率降低到 10 秒级以上 |
| 用户回来恢复 | 重新检测到人脸后，1-2 次采样内恢复活跃状态 |
| 低电量降级 | 电量低于阈值后自动关闭高频视觉 |
| 发热降级 | 设备热状态异常时降低摄像头和动画负载 |
| UI 不白屏 | 降级期间表情保持 idle/sleepy，而不是卡死 |


### 10A.4 视觉识别层

视觉识别层负责将摄像头图像转换为可理解的信息。

MVP 阶段不建议直接上复杂多模态大模型，而应优先实现轻量本地视觉能力。

#### 10A.4.1 MVP 视觉能力

| 能力 | 技术原理 | 输出 |
|---|---|---|
| 人脸存在检测 | 检测画面中是否有人脸 | user_present |
| 人脸数量 | 统计检测到的人脸数 | face_count |
| 人脸位置 | 根据人脸框中心点判断左/中/右 | face_position |
| 距离估计 | 根据人脸框面积占比估算远近 | distance |
| 光线判断 | 分析画面平均亮度 | ambient_light |
| 注视粗判断 | 根据人脸朝向粗略判断 | looking_at_robot |

#### 10A.4.2 人脸位置判断

通过人脸框中心点在画面中的横向位置判断用户在左侧、中间还是右侧。

示例规则：

```text
face_center_x < 0.35          → left
0.35 <= face_center_x <= 0.65 → center
face_center_x > 0.65          → right
```

输出示例：

```json
{
  "face_position": "left"
}
```

该结果可直接驱动机器人眼睛朝左看。

#### 10A.4.3 距离估计

通过人脸框面积占画面面积的比例判断用户距离。

示例规则：

```text
face_area_ratio > 0.28  → near
0.10 - 0.28             → mid
< 0.10                  → far
```

输出示例：

```json
{
  "distance": "near"
}
```

用户靠近时，机器人可以眼睛放大、表情变得惊讶或亲近；用户远离时，机器人可以降低主动发言频率。

#### 10A.4.4 光线判断

通过图像平均亮度判断环境光。

示例规则：

```text
average_luminance < 40   → dark
40 - 180                 → normal
> 180                    → bright
```

输出示例：

```json
{
  "ambient_light": "dark"
}
```

当环境变暗时，机器人可进入夜间或困倦表情，并在合适时提醒用户开灯。

---

### 10A.5 视觉状态模型 VisionState

视觉识别结果不应直接散落在各模块中，而应统一封装为 `VisionState`。

推荐数据结构：

```json
{
  "enabled": true,
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "mid",
  "looking_at_robot": true,
  "ambient_light": "normal",
  "user_emotion_hint": null,
  "confidence": 0.86,
  "last_seen_at": "2026-06-06T10:00:00"
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| enabled | bool | 视觉功能是否开启 |
| user_present | bool | 是否检测到用户 |
| face_count | int | 人脸数量 |
| face_position | enum | left / center / right |
| distance | enum | near / mid / far |
| looking_at_robot | bool | 是否可能看向机器人 |
| ambient_light | enum | dark / normal / bright |
| user_emotion_hint | enum/null | 用户表情倾向 |
| confidence | float | 当前视觉识别置信度 |
| last_seen_at | datetime | 最近一次看到用户的时间 |

### 10A.6 视觉事件模型 VisionEvent

机器人不应持续把每一帧视觉结果都交给 AI，而应在状态变化时发布视觉事件。

例如：

#### 用户出现

```json
{
  "event_type": "vision.user_present",
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "near",
  "timestamp": "2026-06-06T10:00:00"
}
```

#### 用户离开

```json
{
  "event_type": "vision.user_absent",
  "duration_seconds": 60,
  "timestamp": "2026-06-06T10:01:00"
}
```

#### 用户靠近

```json
{
  "event_type": "vision.user_approaching",
  "distance": "near",
  "face_position": "center"
}
```

#### 用户位置变化

```json
{
  "event_type": "vision.face_position_changed",
  "from": "left",
  "to": "center"
}
```

#### 光线变暗

```json
{
  "event_type": "vision.light_changed",
  "ambient_light": "dark"
}
```

这些事件可以进入 Event Bus，再由表情系统、情绪引擎和对话系统分别处理。

### 10A.7 视觉能力分级规划

视觉能力建议按 Level 1 到 Level 5 逐步建设。

#### Level 1：看见有人

这是 MVP 最核心能力。

功能：

- 判断画面中有没有人；
- 判断人脸大致位置；
- 判断用户距离远近。

技术流程：

```text
Camera Frame
    ↓
Face Detection
    ↓
face_count / bounding_box
    ↓
user_present / face_position / distance
```

输出示例：

```json
{
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "mid"
}
```

典型场景：

- 用户出现在手机前，机器人自动睁眼；
- 用户移动到手机左侧，眼睛跟随向左看；
- 用户靠近手机，机器人眼睛放大。

#### Level 2：看见用户状态

在 Level 1 基础上增加连续状态判断。

功能：

- 用户是否靠近；
- 用户是否离开；
- 用户是否回到画面；
- 用户是否长时间在画面中；
- 用户是否大致面向手机；
- 当前环境是否过暗。

技术流程：

```text
当前帧人脸位置
    +
上一帧人脸位置
    +
人脸框大小变化
    +
环境亮度变化
    ↓
判断用户靠近、离开、回来、转头、光线变化
```

典型场景：

- 用户坐到桌前，机器人说“准备开始工作了吗？”
- 用户离开一段时间后回来，机器人说“你回来啦。”
- 光线变暗时，机器人提醒“要不要开个灯？”

#### Level 3：看见表情

增加基础表情倾向识别。

功能：

- 可能开心；
- 可能疲惫；
- 可能惊讶；
- 可能皱眉；
- 中性状态。

技术选择：

```text
ML Kit Face Detection 表情概率
MediaPipe Face Mesh
轻量表情分类模型
云端多模态模型
```

注意：表情识别容易误判，输出必须保守。

推荐输出：

```json
{
  "user_emotion_hint": "possibly_tired",
  "confidence": 0.62
}
```

推荐回复：

> 感觉你可能有点累，要不要先休息一下？

避免回复：

> 你现在很疲惫。

#### Level 4：看见物体

机器人可以识别用户举到镜头前的物品。

功能：

- 识别常见物体；
- 用户举物提问；
- 识别书、本子、杯子、药盒、食品、屏幕内容等；
- 支持视觉问答。

技术流程：

```text
Camera Snapshot
    ↓
Vision Model / Multimodal LLM
    ↓
Object Description
    ↓
LLM Response
```

典型场景：

用户拿起一个东西问：

> 你知道这是什么吗？

机器人进入“认真看”的表情，并回答：

> 我看到你拿着一个杯子，看起来像是不锈钢保温杯。

#### Level 5：看见环境

机器人可以理解所处环境。

功能：

- 判断办公室、卧室、厨房、车内等场景；
- 判断光线是否适合工作；
- 判断桌面是否有人；
- 判断是否长时间无人；
- 判断用户是否可能在休息。

技术组合：

```text
图像场景分类
环境光
时间
用户位置
历史行为
麦克风环境声
```

典型场景：

晚上房间光线较暗，用户仍在工作，机器人提醒：

> 现在光线有点暗，长时间看屏幕可能会累，要不要开个灯？

### 10A.8 视觉与表情系统联动

视觉识别结果应直接驱动机器人屏幕表情，形成“它真的在看我”的体验。

| 视觉事件 | 屏幕反应 |
|---|---|
| 检测到用户出现 | 眼睛睁大，进入欢迎表情 |
| 用户在左侧 | 眼睛向左看 |
| 用户在右侧 | 眼睛向右看 |
| 用户靠近 | 眼睛放大，表情惊讶或亲近 |
| 用户离开 | 眼睛左右看一下，然后进入等待 |
| 光线变暗 | 进入困倦或夜间表情 |
| 多人出现 | 眼睛左右扫视 |
| 用户长时间不看屏幕 | 机器人减少主动发言 |

最重要的 MVP 效果是：

```text
人脸位置变化 → 眼睛方向变化
人脸距离变化 → 眼睛大小变化
用户出现/离开 → 表情状态变化
```

只要这三项做好，机器人的“生命感”会明显增强。

### 10A.9 视觉与对话系统联动

视觉信息可以作为上下文传给 LLM，但不建议把视频流持续传给 LLM。

推荐将视觉状态以结构化摘要传入：

```json
{
  "vision_context": {
    "user_present": true,
    "face_position": "center",
    "distance": "near",
    "ambient_light": "dark",
    "user_emotion_hint": "possibly_tired"
  }
}
```

LLM 基于该上下文生成回复：

```json
{
  "text": "感觉你可能有点累，要不要先休息一下？",
  "expression": "caring",
  "voice": {
    "style": "gentle"
  },
  "haptic": "soft_pulse"
}
```

#### 保守表达原则

视觉判断尤其是表情判断存在误差，因此机器人回复必须使用保守语气。

推荐表达：

```text
我感觉你可能有点累。
看起来你可能有些困。
我不太确定，但你似乎有点卡住了。
```

避免表达：

```text
你现在很疲惫。
你一定很难过。
你生气了。
```

### 10A.10 视觉与情绪引擎联动

视觉事件会影响机器人自身情绪状态。

| 视觉事件 | 情绪变化 |
|---|---|
| 用户出现 | attention +10，mood=happy/neutral |
| 用户靠近 | curiosity +5，mood=surprised |
| 用户离开 | attention -10，sleepiness +5 |
| 用户回来 | mood=happy，attention +10 |
| 光线变暗 | sleepiness +10 |
| 用户可能疲惫 | mood=caring |
| 多人出现 | curiosity +10 |

示例状态更新：

```json
{
  "event_type": "vision.user_returned",
  "state_update": {
    "mood": "happy",
    "attention_delta": 10,
    "sleepiness_delta": -5
  }
}
```

### 10A.11 具体场景设计

#### 场景 1：用户靠近，机器人醒来

用户行为：

用户走到桌面手机前。

视觉判断：

```json
{
  "user_present": true,
  "distance": "near",
  "face_position": "center"
}
```

机器人反应：

- 眼睛从半闭变成睁开；
- 眼球看向用户；
- 嘴巴微笑；
- 可选语音：“你来啦。”

产品价值：

让用户感觉机器人不是被动 App，而是真的注意到了自己。

---

#### 场景 2：眼睛跟随用户移动

用户行为：

用户从手机左侧移动到右侧。

视觉判断：

```text
face_center_x < 0.35          → left
0.35 <= face_center_x <= 0.65 → center
face_center_x > 0.65          → right
```

机器人反应：

- 用户在左侧，眼睛看左；
- 用户在右侧，眼睛看右；
- 用户靠近，眼睛变大；
- 用户远离，眼睛恢复默认。

产品价值：

这是最简单但最强的“拟生命感”来源之一。

---

#### 场景 3：用户离开，机器人进入等待

用户行为：

用户离开座位。

视觉判断：

```json
{
  "event_type": "vision.user_absent",
  "duration_seconds": 30
}
```

机器人反应：

- 眼睛左右看一下；
- 表情变成等待；
- 一段时间后进入困倦或待机；
- 不主动频繁呼唤用户。

可选语音：

> 我先安静待着。

产品价值：

降低能耗，同时让机器人状态更自然。

---

#### 场景 4：用户回来，机器人欢迎

用户行为：

用户离开一段时间后重新坐回手机前。

视觉判断：

```json
{
  "event_type": "vision.user_returned",
  "absent_duration_seconds": 600
}
```

机器人反应：

- 眼睛睁大；
- 嘴巴微笑；
- 轻微震动；
- 语音：“你回来啦，要继续刚才的事吗？”

产品价值：

形成陪伴连续性。

---

#### 场景 5：用户看起来疲惫

用户行为：

用户坐在手机前说：

> 今天效率好低。

视觉判断：

```json
{
  "user_present": true,
  "user_emotion_hint": "possibly_tired",
  "confidence": 0.61,
  "ambient_light": "dark"
}
```

机器人反应：

- 切换关心表情；
- 语速变慢；
- 回复：“感觉你可能有点累。要不要先定一个很小的目标，比如只做 10 分钟？”

注意：

不要绝对判断用户情绪，避免误判造成反感。

---

#### 场景 6：用户举起物品提问

用户行为：

用户拿起一个东西对着手机说：

> 你知道这是什么吗？

技术流程：

```text
Camera Snapshot
    ↓
Multimodal Vision Model
    ↓
Object Caption
    ↓
LLM Response
```

机器人反应：

- 屏幕进入“认真看”的表情；
- 眼睛放大或聚焦；
- 回复：“我看到你拿着一个杯子，看起来像是不锈钢保温杯。”

产品价值：

从“聊天机器人”升级为“有视觉理解的机器人”。

---

#### 场景 7：儿童学习互动

用户行为：

孩子拿着单词卡片问：

> 这个英文怎么读？

技术判断：

视觉模型识别卡片文字。

机器人反应：

- 读出单词；
- 鼓励孩子跟读；
- 根据语音识别判断发音；
- 表情开心鼓励。

产品价值：

可扩展为儿童学习陪伴场景。

---

#### 场景 8：老人陪伴与安全提醒

用户行为：

老人坐在手机前聊天，随后长时间离开画面。

视觉判断：

```json
{
  "user_absent_duration": 3600,
  "last_interaction": "1 hour ago"
}
```

机器人反应：

本地先不频繁打扰，必要时结合授权提醒机制：

> 已经一小时没看到你了，需要我提醒家人确认一下吗？

注意：

该场景涉及安全和隐私，必须有用户或家属授权，不能默认监控。

### 10A.13 后端与 AI Gateway 扩展

MVP 阶段视觉高频任务应在手机端本地完成，不需要全部发送到 Gateway。

只有以下情况需要调用 Gateway 或多模态模型：

1. 用户明确说“你看看这个”；
2. 用户举起物品提问；
3. 用户要求分析图片、纸张、屏幕内容；
4. 手机端轻量模型无法判断；
5. 需要结合视觉结果进行复杂对话。

新增接口建议：

```http
POST /vision/analyze
```

请求：

```json
{
  "session_id": "sess_001",
  "image": "base64_or_file_url",
  "question": "你知道这是什么吗？",
  "vision_context": {
    "user_present": true,
    "distance": "near"
  }
}
```

响应：

```json
{
  "description": "用户手中拿着一个不锈钢保温杯",
  "reply_text": "我看到你拿着一个杯子，看起来像是不锈钢保温杯。",
  "expression": "focused",
  "voice": {
    "style": "neutral",
    "speed": 1.0
  }
}
```

### 10A.14 隐私与安全要求

视觉系统是强隐私功能，必须单独设计安全机制。

#### 摄像头默认关闭

首次进入 App 时，不应默认开启摄像头。

应由用户主动开启，并展示说明：

```text
开启视觉感知后，机器人可以通过前置摄像头判断你是否在画面中、距离远近和大致位置。
默认不会保存视频或照片。
你可以随时关闭该功能。
```

#### 本地优先

以下能力应优先本地处理：

- 是否有人；
- 人脸位置；
- 用户远近；
- 光线强弱；
- 是否离开；
- 是否靠近。

#### 不保存原始画面

除非用户明确要求，否则不保存摄像头原始图片或视频。

#### 明确视觉状态

当摄像头视觉感知开启时，界面必须有明确提示，例如：

```text
视觉感知已开启
```

或显示一个小眼睛图标。

#### 用户主动触发才上传图片

只有用户明确表达：

```text
你看看这个
帮我看一下
这是什么
帮我读一下这张纸
```

才允许抓取当前画面并发送给多模态模型。

#### 表情判断必须保守

机器人不得对用户心理状态做绝对判断，尤其不能基于视觉单独判断用户健康、情绪或意图。

### 10A.15 MVP 推荐实现范围

第一版“看”的功能建议只做三项：

#### 1. 人脸存在检测

判断用户是否在画面中。

用途：

- 用户出现时唤醒；
- 用户离开时待机；
- 用户回来时欢迎。

#### 2. 人脸位置检测

判断用户在画面左、中、右。

用途：

- 眼睛跟随用户移动；
- 形成强烈拟生命感。

#### 3. 人脸距离估计

根据人脸框大小估算用户距离。

用途：

- 靠近时眼睛变大；
- 远离时减少主动发言；
- 贴近时进入小声说话或害羞表情。

MVP 视觉输出可以简化成：

```json
{
  "user_present": true,
  "face_position": "left",
  "distance": "near",
  "ambient_light": "normal"
}
```

只要实现上述三项，就可以让机器人具备明显的“看见用户”的体验。

### 10A.16 视觉功能验收标准

MVP 阶段视觉功能验收标准如下：

| 验收项 | 标准 |
|---|---|
| 人脸存在检测 | 用户进入画面 2 秒内识别 |
| 用户离开检测 | 用户离开 5-10 秒内进入等待 |
| 眼睛跟随 | 用户左右移动时眼睛方向变化 |
| 距离反应 | 用户靠近时眼睛放大或表情变化 |
| 光线判断 | 暗光环境下进入夜间/困倦状态 |
| 隐私开关 | 用户可随时关闭视觉感知 |
| 上传限制 | 未经用户主动触发，不上传图片 |
| 功耗控制 | 桌面模式下低频检测，不明显发热 |
| 误判处理 | 视觉不确定时不输出绝对判断 |

### 10A.17 小结

机器人实现“看”的本质，是将手机摄像头采集到的图像，经过视觉识别算法转换成结构化视觉状态，再驱动机器人表情、语音、情绪和对话。

最小可行路径是：

```text
前置摄像头
    ↓
人脸检测
    ↓
用户是否在场 / 位置 / 距离
    ↓
眼睛跟随 / 欢迎 / 待机 / 靠近反应
```

第一版不需要复杂视觉大模型。只要实现：

```text
看见有人
看见用户在哪边
看见用户靠近或离开
```

就已经能显著提升机器人的生命感和陪伴感。
