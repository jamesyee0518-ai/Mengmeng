## 1. 项目概述

### 1.1 项目背景

当前大多数 AI 聊天产品仍然停留在“文本窗口”或“语音助手”阶段，用户与 AI 的交互主要依赖输入框、语音按钮和消息气泡。这种交互形式功能强，但缺少身体感、陪伴感和情绪反馈。

本项目希望利用智能手机已有的硬件能力，将手机转化为一个具备“看、听、说、表情、触碰反馈和运动感知”的小型具身情感聊天机器人。

在这个设计中：

- 手机屏幕是机器人的脸；
- 前置摄像头是机器人的眼睛；
- 麦克风是机器人的耳朵；
- 扬声器是机器人的嘴巴；
- 触屏是机器人的皮肤；
- 陀螺仪和加速度计是机器人的身体感知；
- 震动马达是机器人的触觉反馈；
- LLM、多模态模型和情绪引擎是机器人的大脑。

### 1.2 产品定义

本产品是一款运行在手机上的具身情感聊天机器人 App。

用户可以将手机放在桌面、床头、车内或随身携带。机器人通过屏幕显示极简眼睛和嘴巴，通过摄像头感知用户，通过麦克风听取语音，通过扬声器进行语音回复，并根据用户触摸、摇晃、拿起、靠近、翻转等行为做出拟人化反应。

产品核心不是单纯问答，而是建立一种“手机里的 AI 伙伴”体验。

### 1.3 一句话定位

把手机变成一个会看、会听、会说、会眨眼、会被触摸回应的 AI 情感伙伴。

### 1.4 核心价值

1. 降低 AI 交互门槛  
   用户不需要打开聊天窗口输入文字，可以像和一个小伙伴说话一样自然交流。

2. 增强陪伴感  
   通过表情、声音、触摸、震动、动作反应形成拟生命感。

3. 复用手机硬件  
   不需要额外购买机器人硬件，旧手机也可以变成桌面 AI 伙伴。

4. 支持本地模型和云端模型混合  
   可根据用户隐私、成本和模型能力需求灵活部署。

5. 可扩展为学习助手、工作助手、老人陪伴、儿童陪伴、桌面宠物等多种场景。

## 2. 产品目标

### 2.1 MVP 目标

MVP 阶段目标是验证“手机具身 AI 伙伴”的核心体验是否成立。

MVP 需要实现：

- 手机屏幕显示简化眼睛和嘴巴；
- 用户可以语音唤醒或点击唤醒；
- 机器人可以听懂语音并进行语音回复；
- 回复时屏幕嘴巴同步动效；
- 机器人具备基础情绪表情；
- 用户点击、长按、摇晃、拿起、倒扣手机时机器人有对应反应；
- 机器人可以根据低电量、充电、夜间等设备状态表现不同情绪；
- 支持基础人格 Prompt；
- 支持结构化 AI 输出控制表情、语音和震动；
- 支持桌面常亮模式。

### 2.2 中期目标

中期目标是增强机器人的感知能力和陪伴能力。

需要实现：

- 前置摄像头人脸检测；
- 用户是否在场检测；
- 用户距离判断；
- 基础表情识别；
- 简单记忆系统；
- 主动提醒；
- 专注陪伴模式；
- 夜间低打扰模式；
- 本地 AI Gateway；
- 本地模型与云端模型自动路由。

### 2.3 长期目标

长期目标是形成一个可持续成长的手机具身机器人平台。

包括：

- 多人格系统；
- 长期记忆与关系成长；
- 多模态视觉问答；
- 物体识别；
- 日程、邮件、任务、智能家居工具调用；
- 儿童安全模式；
- 老人关怀模式；
- 跨设备同步；
- App Store / Android 应用市场正式上线；
- 后续可扩展到平板、桌面屏、车载屏和智能音箱。

## 3. 目标用户

### 3.1 核心用户

1. AI 爱好者  
   喜欢尝试新型 AI 交互方式，希望把旧手机变成 AI 桌面机器人。

2. 独居年轻人  
   希望获得轻量陪伴、情绪回应、日常聊天和生活提醒。

3. 学生和办公人群  
   希望获得学习陪伴、专注提醒、任务拆解和情绪鼓励。

4. 家庭用户  
   希望给儿童或老人提供一个低成本的语音互动陪伴设备。

5. 开发者和极客  
   希望本地部署 LLM，并通过手机获得更生动的交互终端。

### 3.2 典型场景

#### 场景一：桌面 AI 伙伴

用户将手机横放在电脑旁。屏幕上显示一双眼睛。用户靠近时，眼睛看向用户；用户说话时，机器人进入倾听状态；回复时嘴巴动起来，并通过语音回答。

#### 场景二：工作陪伴

用户工作 45 分钟后，机器人轻声提醒：

> 你已经专注很久了，要不要起来活动一下？我可以等你回来。

#### 场景三：情绪陪伴

用户说：

> 今天有点累。

机器人判断用户情绪偏低，切换成关心表情，用温柔语气回复：

> 听起来你今天消耗挺大。要不要我陪你简单复盘一下，还是先安静陪你一会儿？

#### 场景四：宠物式互动

用户摇晃手机，机器人眼睛变成晕圈，并说：

> 哇，别晃啦，我有点头晕。

用户给手机充电，机器人闭眼微笑：

> 补充能量中，我感觉好多啦。

## 7A. 重点章节：机器人通过显示器表达表情的实现技术细节与场景设计

### 7A.1 显示器表情的功能定义

本项目中的“显示器表情”，不是简单播放几张表情图片，而是让手机屏幕成为机器人的“脸”。

这张脸由多个可动画化元素组成：

```text
眼睛
嘴巴
眉毛，可选
脸部背景
光效
粒子
字幕
状态提示
```

它需要与语音、摄像头、触摸、传感器、情绪引擎联动。

完整链路如下：

```text
用户输入 / 摄像头事件 / 传感器事件 / 设备状态
    ↓
情绪引擎判断当前情绪
    ↓
选择表情状态
    ↓
表情状态映射为眼睛、嘴巴、眉毛、背景参数
    ↓
屏幕动画渲染
    ↓
与 TTS、震动、嘴型同步
```

例如，用户摇晃手机时：

```text
motion.shake
    ↓
emotion = dizzy
    ↓
expression = dizzy
    ↓
眼睛变成晕圈
嘴巴变成波浪形
屏幕轻微抖动
震动反馈
语音说：“哇，别晃啦。”
```

这才是“显示器表情系统”的完整意义。

---

### 7A.2 表情系统设计目标

#### 7A.2.1 让机器人看起来“活着”

即使用户没有说话，机器人也应该有轻微生命感，例如：

```text
慢速眨眼
眼睛轻微呼吸缩放
视线偶尔移动
背景轻微渐变
嘴巴偶尔微笑
```

这样可以让它不像一个静态 App，而像一个正在陪伴用户的小生命。

#### 7A.2.2 让用户理解机器人状态

用户应该通过表情快速判断机器人当前状态。

| 表情 | 用户理解 |
|---|---|
| 睁眼看着用户 | 我在 |
| 眼睛左右转 | 我在思考 |
| 半闭眼 | 我困了 / 夜间模式 |
| 晕圈眼 | 我被晃晕了 |
| 闭眼微笑 | 我在充电 / 很舒服 |
| 眼睛变大 | 我惊讶 / 我看见你了 |
| 嘴巴小幅开合 | 我正在说话 |
| 嘴巴闭合 | 我在听 / 安静 |

#### 7A.2.3 让情绪和交互更自然

同一句话配不同表情，用户感受完全不同。

例如机器人说：

> 我在呢。

不同表情带来的感受：

| 表情 | 感受 |
|---|---|
| 默认睁眼 | 正常回应 |
| 关心表情 | 温柔陪伴 |
| 困倦表情 | 夜间陪伴 |
| 开心表情 | 欢迎用户 |
| 晕圈表情 | 被摇晃后的俏皮反应 |

### 7A.3 技术实现总体架构

表情系统建议分为六层：

```text
Expression Event 输入层
    ↓
Expression State 表情状态层
    ↓
Expression Mapper 参数映射层
    ↓
Animation Controller 动画控制层
    ↓
Rendering Layer 渲染层
    ↓
Sync Layer 联动层
```

各层职责如下：

| 层级 | 职责 |
|---|---|
| 输入层 | 接收对话、视觉、语音、触摸、传感器事件 |
| 状态层 | 维护当前表情状态 |
| 映射层 | 将 emotion 转换为眼睛、嘴巴、背景参数 |
| 动画层 | 控制眨眼、嘴型、过渡、弹跳 |
| 渲染层 | 使用 Flutter / Rive / Lottie 绘制 |
| 联动层 | 与 TTS、震动、摄像头、传感器同步 |

### 7A.4 推荐技术路线

#### 7A.4.1 Flutter CustomPainter

MVP 最推荐使用 Flutter 的 `CustomPainter` 自绘眼睛、嘴巴和背景。

优点：

```text
性能好
可控性强
不用依赖大量素材
易于参数化
适合做极简表情
跨平台一致性较好
```

适合实现：

```text
眼睛形状变化
眼球方向变化
嘴巴开合
嘴巴弧线变化
背景呼吸动画
眨眼动画
屏幕轻微抖动
```

#### 7A.4.2 Rive 动画

Rive 适合做更精细的状态机动画。

优点：

```text
动画师可参与
状态机友好
适合做复杂表情过渡
运行时可调参数
```

适合后续版本：

```text
复杂眼神
可爱表情
状态机动画
品牌化角色
多套皮肤
```

#### 7A.4.3 Lottie 动画

Lottie 适合播放预制动画。

优点：

```text
接入简单
视觉效果好
适合一次性动效
```

适合：

```text
充电动画
低电量动画
惊讶动画
庆祝动画
睡眠动画
```

缺点是交互参数化能力不如 CustomPainter 和 Rive。

#### 7A.4.4 Unity / 3D Avatar

不建议 MVP 使用 Unity 或 3D Avatar。

原因：

```text
开发成本高
性能消耗大
和手机桌面常亮场景不匹配
过早复杂化
```

如果未来要做高级数字人版本，再考虑 Unity 或 3D 引擎。

### 7A.9 表情优先级机制

不同事件可能同时触发表情，例如机器人正在说话时手机低电量，或者用户摇晃手机时机器人正在回答。

因此需要设计表情优先级。

推荐优先级：

| 优先级 | 表情来源 | 示例 |
|---|---|---|
| P0 | 安全/系统状态 | 来电、权限错误、严重低电量 |
| P1 | 强传感器事件 | 摇晃、倒扣、拿起 |
| P2 | 语音播放状态 | speaking、interrupted |
| P3 | 用户主动对话情绪 | caring、happy、confused |
| P4 | 视觉事件 | 用户出现、靠近、离开 |
| P5 | 默认情绪状态 | neutral、sleepy、focus |
| P6 | 随机微表情 | 眨眼、视线游走 |

示例：

```text
正在 speaking
    +
用户强烈摇晃手机
    ↓
临时切换 dizzy
    ↓
说完“别晃啦”
    ↓
回到 neutral 或 interrupted
```

---

### 7A.10 动画过渡机制

表情切换不能突兀，应该有过渡。

#### 7A.10.1 基础过渡

```text
neutral → listening：眼睛睁大，嘴巴闭合
listening → thinking：眼睛左右移动
thinking → speaking：嘴巴开始开合
speaking → neutral：嘴巴闭合，回到微笑
```

#### 7A.10.2 过渡参数

```json
{
  "from": "neutral",
  "to": "happy",
  "transition": "smooth",
  "duration_ms": 300,
  "easing": "easeOut"
}
```

#### 7A.10.3 特殊过渡

| 过渡 | 表现 |
|---|---|
| neutral → surprised | 快速放大眼睛 |
| surprised → happy | 眼睛变月牙，嘴巴微笑 |
| neutral → sleepy | 逐渐半闭眼，降低亮度 |
| speaking → interrupted | 嘴巴立刻闭合，眼睛一怔 |
| neutral → dizzy | 屏幕震动，眼睛旋转 |
| low_battery → charging | 从委屈变成闭眼微笑 |

### 7A.11 自然生命感：微表情系统

即使没有用户输入，机器人也不能完全静止。

#### 7A.11.1 自动眨眼

```text
每 3-7 秒随机眨眼一次
困倦状态眨眼更慢
惊讶状态眨眼更少
开心状态眨眼更轻快
```

#### 7A.11.2 呼吸动画

```text
眼睛或背景轻微缩放
周期 2-4 秒
幅度很小
```

#### 7A.11.3 视线游走

```text
空闲时偶尔看左、看右
检测到用户时看向用户
思考时眼睛左右移动
```

#### 7A.11.4 情绪残留

表情不要立刻完全回到默认。

例如：

```text
用户夸奖机器人
    ↓
happy 维持 2 秒
    ↓
soft_happy 维持 5 秒
    ↓
neutral
```

这样机器人显得更自然。

### 7A.12 具体场景设计

#### 场景 1：默认待机

状态：

```text
用户没有说话
手机放在桌面
机器人处于 idle
```

表情：

```text
眼睛正常睁开
每隔几秒自然眨眼
嘴巴轻微微笑
背景轻微呼吸
```

价值：

让手机不是静态界面，而是有“在场感”。

---

#### 场景 2：用户唤醒

用户说：

> 小眨。

或双击屏幕。

表情变化：

```text
sleepy / neutral
    ↓
眼睛快速睁开
    ↓
看向用户
    ↓
嘴巴微笑
```

机器人说：

> 我在呢，你说。

---

#### 场景 3：正在倾听

用户长按屏幕或语音唤醒后开始说话。

表情：

```text
眼睛专注看向用户
嘴巴闭合
背景轻微脉冲
字幕显示“我在听”
```

如果用户停顿太久：

```text
眼睛轻轻眨一下
嘴巴仍闭合
不急着打断
```

---

#### 场景 4：正在思考

用户说完问题后，机器人等待 LLM 回复。

表情：

```text
眼睛左右移动
嘴巴平直
背景有轻微流动
可显示“想一想……”
```

如果等待时间超过 3 秒：

```text
眼睛稍微眯一下
显示“这个问题我需要想一下”
```

价值：

减少模型延迟带来的等待焦虑。

---

#### 场景 5：正在说话

TTS 开始播放。

表情：

```text
眼睛看向用户
嘴巴随语音开合
根据语气调整嘴巴幅度
字幕同步显示
```

如果是温柔语气：

```text
眼睛柔和
嘴巴开合幅度小
背景暖色
```

如果是开心语气：

```text
月牙眼
嘴巴上扬
整体轻微弹跳
```

---

#### 场景 6：用户靠近

视觉系统检测到用户靠近。

表情：

```text
眼睛变大
眼神聚焦
嘴巴微笑
```

机器人可以说：

> 你靠近啦，我看到你了。

或者不说话，只通过眼神回应。

---

#### 场景 7：用户离开

视觉系统检测到用户离开画面。

表情：

```text
眼睛左右看一下
嘴巴变成小平嘴
等待几秒
进入 sleepy 或 idle
```

如果用户长时间离开：

```text
眼睛半闭
背景变暗
进入省电状态
```

---

#### 场景 8：用户回来

用户重新出现在画面中。

表情：

```text
眼睛快速睁大
嘴巴微笑
背景轻微亮起
```

机器人说：

> 你回来啦。

如果之前有未完成任务，可以说：

> 要继续刚才的事吗？

---

#### 场景 9：被摇晃

加速度计检测到强烈摇晃。

表情：

```text
眼睛变成旋涡
嘴巴波浪形
屏幕轻微抖动
背景晃动
```

机器人说：

> 哇，别晃啦，我有点头晕。

说完后：

```text
dizzy 维持 2 秒
逐渐恢复 neutral
```

---

#### 场景 10：充电

检测到充电插入。

表情：

```text
闭眼微笑
背景呼吸发光
可显示能量条
```

机器人说：

> 补充能量中，我感觉好多啦。

---

#### 场景 11：低电量

电量低于阈值。

表情：

```text
眼睛半闭
嘴巴下弯
背景变暗
动画速度降低
```

机器人说：

> 我快没电了，能不能帮我充一下？

如果用户不处理，减少语音输出，只保留低电量提示。

---

#### 场景 12：夜间模式

时间较晚或光线很暗。

表情：

```text
眼睛半闭
背景低亮度
嘴巴动作变慢
字幕亮度降低
```

语音：

```text
音量降低
语速变慢
回复变短
```

价值：

避免夜间打扰。

---

#### 场景 13：用户夸奖机器人

用户说：

> 你真聪明。

表情：

```text
眼睛变成月牙
嘴巴上扬
整体轻微弹跳
背景轻微发光
```

机器人说：

> 嘿嘿，被你夸我有点开心。

注意不要过度拟人化或情感绑架。

---

#### 场景 14：识别失败

STT 或 LLM 没有理解用户。

表情：

```text
一只眼稍大
一只眼稍小
嘴巴变成小波浪
```

机器人说：

> 我刚刚没太听清，可以再说一遍吗？

价值：

用表情降低失败带来的挫败感。

---

#### 场景 15：用户要求安静

用户说：

> 别说话了。

表情：

```text
嘴巴闭合
眼睛轻轻眨一下
进入安静模式
```

机器人只显示字幕：

> 好，我安静陪着。

后续不再主动语音输出。

### 7A.15 性能与功耗设计

表情系统会长期显示，因此必须关注性能和耗电。

#### 7A.15.1 帧率策略

| 模式 | 帧率 |
|---|---|
| 正常交互 | 30-60 fps |
| 桌面待机 | 15-30 fps |
| 夜间模式 | 10-15 fps |
| 低电量模式 | 5-10 fps |
| 静态休眠 | 几乎停止动画 |

#### 7A.15.2 动画节流

不需要每一帧都重绘所有元素。

可以分层渲染：

```text

眼睛中频更新
嘴巴说话时高频更新
字幕按文本更新
```

#### 7A.15.3 防 OLED 烧屏

如果长时间桌面常亮，需要防止 OLED 烧屏。

建议：

```text
整体脸部位置每隔几分钟轻微移动
背景亮度不要过高
避免长时间显示纯白高亮元素
夜间自动降低亮度
提供屏幕保护模式
```

#### 7A.15.4 微表情基线抖动机制

为了同时解决 OLED 烧屏和表情呆板问题，表情系统应引入“微表情基线抖动”。

核心思想：

- 眼睛、嘴巴、脸部主体不应长期停留在完全相同像素位置；
- 每隔一段时间通过低频正弦波、余弦波或 Perlin Noise 产生 2-5 像素级别的微小偏移；
- 该偏移应足够小，用户几乎感知不到，但能打破 OLED 像素长期固定点亮的问题；
- 该偏移也能让机器人看起来更像有轻微呼吸和生命感。

推荐公式：

```text
X_render = X_base + sin(ω1 × t) × A1
Y_render = Y_base + cos(ω2 × t) × A2
```

参数建议：

| 参数 | 建议值 | 说明 |
|---|---:|---|
| A1 | 2-5 px | 横向最大偏移 |
| A2 | 2-5 px | 纵向最大偏移 |
| ω1 | 0.03-0.08 | 横向低频速度 |
| ω2 | 0.02-0.06 | 纵向低频速度 |
| update_interval | 16-100 ms | 根据当前帧率策略调整 |

伪代码示例：

```dart
final dx = sin(timeSeconds * omegaX) * amplitudeX;
final dy = cos(timeSeconds * omegaY) * amplitudeY;

final renderCenter = Offset(
  baseCenter.dx + dx,
  baseCenter.dy + dy,
);
```

如果使用 Perlin Noise，可获得更自然、不完全周期化的漂移效果：

```text
X_render = X_base + noise(t × k1) × A1
Y_render = Y_base + noise(t × k2 + offset) × A2
```

MVP 推荐：

```text
优先使用正弦 / 余弦低频偏移
振幅控制在 2-3 px
夜间和低电量模式下进一步降低更新频率
```

验收标准：

| 验收项 | 标准 |
|---|---|
| 微位移可用 | 长时间待机时眼睛主体像素位置会缓慢变化 |
| 用户无明显感知 | 不应产生明显晃动或眩晕感 |
| 防烧屏 | 常亮模式下避免核心高亮元素长期固定 |
| 生命感 | 待机时表情略有呼吸感，而非完全静止 |

## 8A. 重点章节：机器人“说”的功能实现技术细节与场景设计

### 8A.1 “说”的功能定义

本项目中的机器人“说”，不是简单把文字通过 TTS 念出来，而是一个完整的多模态表达系统。

完整链路如下：

```text
AI 生成回复内容
    ↓
判断说话情绪和语气
    ↓
选择语音音色、语速、音高、停顿
    ↓
TTS 生成语音
    ↓
手机扬声器播放
    ↓
屏幕嘴巴动画同步开合
    ↓
眼睛、表情、背景和震动共同配合
```

因此，“说”的核心不是单一 TTS 能力，而是：

```text
文本内容 + 语音情绪 + 嘴型动画 + 表情状态 + 场景语气 + 打断控制
```

例如，同一句“我在呢”，在不同场景中应该有不同的表达方式：

| 场景 | 说法 | 表情 |
|---|---|---|
| 用户普通唤醒 | 我在呢，你说。 | 默认睁眼 |
| 用户疲惫 | 我在呢，慢慢说。 | 温柔关心 |
| 用户半夜说话 | 嗯……我在，小声一点陪你。 | 困倦低亮度 |
| 用户摇晃手机 | 哇，别晃啦，我有点头晕。 | 晕圈眼 |
| 充电中 | 正在补充能量，我感觉好多啦。 | 闭眼微笑 |

---

### 8A.2 技术实现总体流程

机器人“说”的技术链路建议分为七层：

```text
Dialogue Response 对话生成
    ↓
Speech Policy 说话策略
    ↓
Voice Style Mapping 语音风格映射
    ↓
TTS Synthesis 语音合成
    ↓
Audio Playback 音频播放
    ↓
Mouth Animation 嘴巴动画
    ↓
Expression & Haptic Sync 表情和震动同步
```

各层职责如下：

| 层级 | 职责 |
|---|---|
| 对话生成层 | LLM 生成回复内容 |
| 说话策略层 | 判断是否应该说、说多长、用什么语气 |
| 语音风格层 | 映射音色、语速、音高、音量、情绪 |
| TTS 层 | 将文本转成语音 |
| 播放层 | 控制扬声器播放、暂停、停止、打断 |
| 嘴型层 | 根据播放状态、音频能量或音素驱动嘴巴动画 |
| 联动层 | 表情、眼神、背景、震动同步变化 |

---

### 8A.3 LLM 回复生成

机器人说话内容来自对话大脑。LLM 不应只返回文本，而应返回结构化结果，让客户端知道：

- 要说什么；
- 是否要说出声；
- 用什么语气说；
- 用什么表情配合；
- 嘴巴如何动；
- 是否震动；
- 是否允许用户打断。

推荐输出格式：

```json
{
  "text": "听起来你今天消耗挺大。要不要我陪你简单复盘一下？",
  "emotion": "concerned",
  "expression": "caring",
  "voice": {
    "style": "gentle",
    "speed": 0.9,
    "pitch": 0.95,
    "volume": 0.8
  },
  "mouth_action": "talking_soft",
  "haptic": "soft_pulse",
  "should_speak": true,
  "allow_interrupt": true
}
```

这样 App 不需要猜测语气，而是直接按照结构化参数执行。

### 8A.4 说话策略 Speech Policy

不是所有 AI 回复都应该被朗读出来。机器人需要根据当前场景判断是否适合发声。

判断因素包括：

- 当前是否处于夜间模式；
- 用户是否开启静音；
- 用户是否处于专注模式；
- 当前是否在公共场所模式；
- 用户是否主动语音提问；
- 回复内容是否过长；
- 设备电量是否过低；
- 用户是否倒扣手机；
- 用户是否说过“安静一点”；
- 当前是否正在通话或播放其他系统音频。

#### 8A.4.1 典型说话策略

| 场景 | 说话策略 |
|---|---|
| 用户主动语音提问 | 默认语音回复 |
| 用户文字输入 | 可只显示字幕，或按设置朗读 |
| 夜间模式 | 降低音量，缩短回复 |
| 专注模式 | 少说话，尽量简短 |
| 低电量 | 减少长语音 |
| 用户倒扣手机 | 不说话，进入睡眠 |
| 用户正在通话 | 不播放语音 |
| 公共场所模式 | 默认字幕，不外放 |
| 用户靠近手机 | 可小声说话 |
| 用户说“安静一点” | 进入低音量或静音模式 |

#### 8A.4.2 策略输出示例

```json
{
  "should_speak": true,
  "speech_mode": "whisper",
  "max_sentence_count": 2,
  "show_subtitle": true,
  "allow_interrupt": true
}
```

### 8A.8 嘴巴动画同步

“说”的体验不能只有声音，屏幕上的嘴巴必须跟随语音动起来。

#### 8A.8.1 MVP：播放状态驱动嘴型

最简单方案是根据 TTS 播放状态驱动嘴巴开合。

流程：

```text
TTS 播放开始
    ↓
进入 speaking 表情
    ↓
嘴巴按模拟节奏开合
    ↓
TTS 播放结束
    ↓
嘴巴闭合，回到 idle
```

嘴巴参数示例：

```json
{
  "mouth_shape": "talking",
  "open_level": 0.6,
  "speed": "normal"
}
```

MVP 阶段可以先不读取真实音频波形，而是用模拟节奏：

```text
短句：嘴巴快速开合 1-2 秒
长句：嘴巴持续开合
停顿：嘴巴微闭
结束：嘴巴闭合微笑
```

优点是实现简单，适合第一版。

#### 8A.8.2 进阶：音频能量驱动

如果可以获取 TTS 音频数据，可分析音频振幅，驱动嘴巴开合。

```text
Audio PCM / waveform
    ↓
计算 RMS 音量
    ↓
映射 mouth_open_level
    ↓
每帧更新嘴巴开合
```

映射示例：

```text
RMS 低 → mouth_open_level = 0.2
RMS 中 → mouth_open_level = 0.5
RMS 高 → mouth_open_level = 0.9
```

这种方式比模拟节奏更自然。

#### 8A.8.2A MVP 推荐：音频振幅平滑法

虽然模拟节奏实现简单，但第一版建议直接采用“音频能量驱动 + 平滑滤波”的嘴型同步方案。原因是模拟节奏容易出现明显“对不上口型”的机械感，用户会觉得机器人在假动嘴。

推荐流程：

```text
TTS 音频播放
    ↓
获取当前播放帧 PCM 数据
    ↓
计算 RMS 均方根振幅
    ↓
归一化为 0-1 的 RMS_normalized
    ↓
一阶低通滤波平滑
    ↓
映射为 mouth_open_level
    ↓
驱动嘴巴开合
```

RMS 计算：

```text
RMS = sqrt((x1² + x2² + ... + xn²) / n)
```

为了防止嘴巴过度闪烁，需要对开合值做一阶低通滤波：

```text
level_new = α × level_old + (1 - α) × RMS_normalized
```

推荐参数：

| 参数 | 建议值 | 说明 |
|---|---:|---|
| α | 0.7 | 平滑系数，越大越稳定 |
| RMS window | 20-50 ms | 每次计算的音频窗口 |
| min_open | 0.05 | 静音时嘴巴最小开合 |
| max_open | 0.85 | 最大张嘴程度 |
| update_rate | 30 fps | 嘴型刷新频率 |

伪代码示例：

```dart
double smoothMouthLevel({
  required double oldLevel,
  required double rmsNormalized,
  double alpha = 0.7,
}) {
  final next = alpha * oldLevel + (1 - alpha) * rmsNormalized;
  return next.clamp(0.05, 0.85);
}
```

Flutter 实现建议：

| 方案 | 说明 |
|---|---|
| flutter_sound | 用于音频播放和 PCM 数据处理，适合自管音频流 |
| 原生平台插件 | iOS/Android 分别获取播放音频能量，效果更稳 |
| 后端 TTS 返回 mouth_timeline | 后端直接生成嘴型时间线，前端按时间播放 |
| 系统 TTS fallback | 无法获取 PCM 时退回模拟嘴型 |

MVP 推荐决策：

```text
优先使用 RMS 音频能量驱动嘴巴
如果系统 TTS 无法暴露 PCM，则使用“播放状态 + 文本长度 + 标点停顿”的模拟嘴型 fallback
```

验收标准：

| 验收项 | 标准 |
|---|---|
| 嘴巴随音量变化 | 声音大时张嘴更大，停顿时嘴巴收小 |
| 无高频抖动 | 嘴巴不会疯狂闪烁 |
| 停顿自然 | 标点或静音段嘴巴接近闭合 |
| fallback 可用 | 无法获取 PCM 时仍能用模拟嘴型说话 |


#### 8A.8.3 高级：音素级口型同步

高级版本可基于音素或 viseme 实现更准确的口型同步。

```text
文本
    ↓
TTS / phoneme aligner
    ↓
音素序列
    ↓
viseme 口型序列
    ↓
嘴巴动画
```

音素和嘴型示例：

| 音素 | 嘴型 |
|---|---|
| a | 张大嘴 |
| o | 圆嘴 |
| i | 横向嘴 |
| m/b/p | 闭嘴 |
| f/v | 咬唇 |
| silence | 闭嘴 |

第一版不建议做音素级口型同步，除非产品已经验证成立并进入高级表现阶段。

### 8A.15A 语音打断的 AEC 与 VAD 工程要求

文档中虽然定义了“用户说停一下即可打断机器人说话”，但实际实现中存在一个关键工程瓶颈：

当手机扬声器正在播放 TTS 时，麦克风会采集到机器人自己的声音。如果没有回声消除和本地语音激活检测，系统很可能把机器人自己的 TTS 当成用户语音，或者完全听不到用户真正说出的“停一下”。

因此，语音打断必须加入 AEC、VAD 和关键词检测的组合策略。

#### 8A.15A.1 核心问题

```text
机器人 TTS 外放
    ↓
扬声器声音进入麦克风
    ↓
麦克风采集到“机器人自己的声音 + 用户声音 + 环境噪声”
    ↓
如果没有 AEC/VAD
    ↓
打断识别失败或误触发
```

典型失败情况：

| 情况 | 结果 |
|---|---|
| TTS 音量较大 | 用户说“停一下”被机器人声音淹没 |
| 无回声消除 | 系统误以为机器人自己的声音是用户输入 |
| 无本地 VAD | 一直上传噪声给 STT，功耗和延迟升高 |
| 云端 STT 延迟过高 | 用户打断不及时，体验失效 |

#### 8A.15A.2 推荐打断链路

```text
TTS speaking
    ↓
开启本地低功耗 VAD
    ↓
AEC 消除扬声器回声
    ↓
检测到用户语音活动
    ↓
本地关键词检测：停一下 / 别说了 / 安静 / pause / stop
    ↓
立即 stop TTS
    ↓
表情切换 interrupted
    ↓
进入 listening 或 quiet mode
```

#### 8A.15A.3 AEC 回声消除

AEC 的目标是从麦克风输入中尽量减去扬声器正在播放的 TTS 音轨。

实现建议：

| 平台 | 实现方向 |
|---|---|
| iOS | 使用 AVAudioSession 的 voiceChat / playAndRecord 模式 |
| Android | 使用 AcousticEchoCanceler、NoiseSuppressor、AutomaticGainControl |
| Flutter | 通过原生插件封装平台 AEC 能力 |
| WebRTC | 使用 WebRTC Audio Processing Module 做 AEC/NS/AGC |

推荐策略：

```text
MVP 阶段至少启用系统级 AEC
正式版考虑引入 WebRTC APM 作为跨平台音频处理层
```

#### 8A.15A.4 VAD 语音激活检测

VAD 用于判断当前麦克风输入中是否有“真人说话”。

要求：

- VAD 必须本地运行；
- VAD 在 TTS 播放期间保持低功耗监听；
- 只有检测到疑似用户语音时，才进入关键词识别或 STT；
- 不能把 TTS 外放本身当作用户说话。

推荐实现：

| 方案 | 说明 |
|---|---|
| WebRTC VAD | 经典轻量方案 |
| Silero VAD | 效果好，但端侧成本略高 |
| 平台原生语音检测 | 接入简单，但能力差异较大 |
| 关键词小模型 | 专门识别“停一下”等打断词 |

#### 8A.15A.5 打断关键词

MVP 建议内置本地打断词：

```text
停
停一下
别说了
安静
闭嘴
暂停
不用说了
stop
pause
quiet
```

用户说出打断词后，不需要等待完整云端 LLM 回复，应立即执行：

```json
{
  "event_type": "speech.interrupted",
  "reason": "local_keyword",
  "action": "stop_tts",
  "next_expression": "interrupted"
}
```

#### 8A.15A.6 降级策略

如果当前设备无法可靠支持 AEC/VAD，则使用备用交互：

| 条件 | 降级方式 |
|---|---|
| AEC 不可用 | 显示明显的“点按可打断”按钮 |
| VAD 不稳定 | 仅支持触摸打断 |
| TTS 外放过大 | 自动降低播放音量 |
| 耳机连接 | 优先支持语音打断，因为回声更少 |
| 公共场所模式 | 默认字幕，不外放 |

MVP 验收标准：

| 验收项 | 标准 |
|---|---|
| 触摸打断 | 点击或长按可立即停止 TTS |
| 关键词打断 | 中等音量下可识别“停一下” |
| 回声抑制 | 机器人自己的 TTS 不应频繁误触发 |
| 降级提示 | 语音打断不可用时显示触摸打断提示 |
| 延迟 | 本地打断响应应小于 300ms |


### 8A.15 MVP 推荐实现范围

第一版“说”的功能建议只做以下内容：

1. 使用系统 TTS；
2. 支持中文语音播放；
3. 支持语速、音量、音高基础控制；
4. 支持播放开始、播放结束回调；
5. 支持用户点击或长按打断；
6. 支持嘴巴在说话时开合；
7. 支持不同情绪映射不同说话风格；
8. 支持夜间降低音量；
9. 支持静音模式；
10. 支持字幕同步显示。

MVP 不建议做：

- 复杂情绪化 TTS；
- 音素级口型同步；
- 语音克隆；
- 多角色实时切换；
- 长文本朗读；
- 后台持续播报。

### 8A.17 小结

机器人“说”的实现，本质上是把 AI 回复转化成一个完整的多模态表达动作。

最小可行链路是：

```text
LLM 生成回复
    ↓
返回 text + voice_style + expression
    ↓
手机端 TTS 播放
    ↓
嘴巴动画同步
    ↓
表情和震动配合
```

MVP 阶段不需要一开始追求高级语音克隆或音素级口型同步。第一版只要实现：

```text
能自然朗读
能根据情绪调整语速和音量
说话时嘴巴会动
用户可以打断
夜间和静音模式可控
```

## 10A. 重点章节：机器人“看”的功能实现原理与场景设计

### 10A.1 “看”的功能定义

本项目中的“看”不是简单打开摄像头显示画面，而是让机器人通过手机摄像头获得对用户、环境和物体的视觉理解，并将视觉结果转化为可驱动表情、语音、对话和行为的结构化事件。

完整链路如下：

```text
摄像头采集画面
    ↓
视觉算法识别画面内容
    ↓
转化为结构化视觉状态与视觉事件
    ↓
输入情绪引擎和对话系统
    ↓
驱动机器人眼神、表情、语音、震动和主动行为
```

例如，用户走到手机前时，不需要说话，机器人通过前置摄像头检测到“有人靠近”，然后屏幕眼睛睁大、视线转向用户，并用语音说：

> 你回来啦。

这类体验是本产品从普通聊天 App 升级为“手机具身情感机器人”的关键。

---

### 10A.2 技术实现总体流程

机器人视觉能力建议分为五层：

```text
Camera Capture 摄像头采集层
    ↓
Vision Detection 视觉识别层
    ↓
Vision State Mapping 视觉状态映射层
    ↓
Vision Event Bus 视觉事件分发层
    ↓
Expression / Dialogue / Emotion 联动层
```

各层职责如下：

| 层级 | 职责 | 输出 |
|---|---|---|
| 摄像头采集层 | 获取前置摄像头图像帧 | camera frame |
| 视觉识别层 | 检测人脸、位置、光线、物体等 | raw detection result |
| 视觉状态映射层 | 将识别结果转为业务状态 | VisionState |
| 视觉事件分发层 | 判断状态变化并发布事件 | VisionEvent |
| 联动层 | 驱动表情、对话、情绪、语音 | expression / reply / action |

### 10A.3 摄像头采集层

手机前置摄像头是机器人的“眼睛”。App 需要通过手机摄像头周期性获取图像帧，用于视觉识别。

Flutter 中可使用：

```text
camera
google_mlkit_face_detection
permission_handler
```

也可以在 iOS 和 Android 端分别使用平台原生能力：

```text
iOS：Vision Framework
Android：ML Kit Face Detection
跨平台：MediaPipe / Google ML Kit Flutter 插件
```

基础流程：

```text
申请摄像头权限
    ↓
打开前置摄像头
    ↓
获取实时图像帧
    ↓
抽帧或降分辨率
    ↓
送入视觉识别模块
```

为了控制功耗和发热，不建议持续高频分析高清视频。

推荐策略：

| 模式 | 视觉检测策略 |
|---|---|
| 普通聊天模式 | 用户唤醒或说话时短时启用 |
| 桌面陪伴模式 | 每 1-3 秒低频检测一次 |
| 专注模式 | 低频检测，只判断用户是否在场 |
| 夜间模式 | 默认关闭或极低频检测 |
| 隐私模式 | 完全关闭摄像头 |
| 用户明确请求“你看看” | 抓取单帧图像进行视觉理解 |

MVP 阶段推荐使用低分辨率图像，例如：

```text
320 × 240
640 × 480
```

这些分辨率已经足够做人脸存在、位置、距离和光线判断。

### 10A.3A 音视频并发下的功耗与发热动态降级策略

将手机作为“桌面常亮、常听、可视觉感知”的设备时，核心瓶颈不是单一算法精度，而是音视频流并发带来的发热、功耗和降频问题。

高风险组合：

```text
前置摄像头持续采集
    +
人脸检测持续运行
    +
麦克风持续唤醒监听 / 降噪 / VAD
    +
屏幕常亮动画渲染
    +
网络请求 / LLM 等待
    ↓
15-30 分钟内可能明显发热、掉帧、降频、耗电
```

因此，视觉系统必须具备动态降级能力，不能只依赖低分辨率。

#### 10A.3A.1 视觉检测动态频率

视觉检测频率应根据用户是否在场、最近交互时间、电量、温度和模式动态调整。

推荐状态机：

| 状态 | 条件 | 检测频率 |
|---|---|---:|
| active_tracking | 用户在画面中，正在互动 | 0.5-1 秒/次 |
| desktop_idle | 用户在画面中，但无交互 | 2-3 秒/次 |
| no_face_recent | 最近 30 秒未检测到人脸 | 5 秒/次 |
| no_face_long | 超过 3 分钟未检测到人脸 | 10-15 秒/次 |
| thermal_save | 设备发热或低电量 | 15-30 秒/次或关闭 |
| privacy/off | 隐私模式或用户关闭 | 不检测 |

推荐滑落逻辑：

```text
检测到人脸
    ↓
active_tracking / desktop_idle
    ↓
连续 30 秒无人脸
    ↓
no_face_recent：每 5 秒检测
    ↓
连续 3 分钟无人脸
    ↓
no_face_long：每 10-15 秒检测
    ↓
检测到人脸
    ↓
立即恢复 active_tracking
```

#### 10A.3A.2 多模态资源调度原则

摄像头、麦克风、屏幕动画不能同时长期满功率运行。

推荐策略：

| 资源 | 降级方式 |
|---|---|
| 摄像头 | 抽帧、低分辨率、无人时降频、发热时关闭 |
| 人脸检测 | 从连续检测改为定时采样 |
| 麦克风 | 本地 VAD 低功耗监听，避免持续云端 STT |
| 屏幕动画 | 待机降到 15-30 fps，低电量降到 5-10 fps |
| 网络 | 只在用户明确输入或重要事件时请求 Gateway |
| LLM | 长时间空闲时不保活重模型连接 |

#### 10A.3A.3 温度与电量触发降级

App 应监听设备电量、充电状态、低电量模式和系统热状态。

推荐规则：

| 条件 | 动作 |
|---|---|
| 电量 < 20% 且未充电 | 降低摄像头频率、降低动画帧率 |
| 电量 < 10% | 关闭视觉检测，仅保留触摸唤醒 |
| 系统进入低电量模式 | 禁用主动视觉，降低屏幕动画 |
| 设备发热 | 降低视觉检测到 15-30 秒/次 |
| 充电且屏幕常亮 | 控制亮度，避免边充边高负载 |

#### 10A.3A.4 端侧调度伪代码

```dart
Duration getVisionInterval(DeviceRuntimeState state) {
  if (!state.visionEnabled || state.privacyMode) {
    return Duration.zero;
  }

  if (state.isThermalHot || state.batteryLevel < 10) {
    return const Duration(seconds: 30);
  }

  if (!state.userPresentForLongTime) {
    return const Duration(seconds: 10);
  }

  if (!state.userPresentRecently) {
    return const Duration(seconds: 5);
  }

  if (state.isUserInteracting) {
    return const Duration(milliseconds: 800);
  }

  return const Duration(seconds: 3);
}
```

#### 10A.3A.5 MVP 验收标准

| 验收项 | 标准 |
|---|---|
| 无人时降频 | 连续无人 30 秒后，检测频率自动降低到 5 秒级 |
| 长时间无人 | 超过 3 分钟无人后，检测频率降低到 10 秒级以上 |
| 用户回来恢复 | 重新检测到人脸后，1-2 次采样内恢复活跃状态 |
| 低电量降级 | 电量低于阈值后自动关闭高频视觉 |
| 发热降级 | 设备热状态异常时降低摄像头和动画负载 |
| UI 不白屏 | 降级期间表情保持 idle/sleepy，而不是卡死 |


### 10A.4 视觉识别层

视觉识别层负责将摄像头图像转换为可理解的信息。

MVP 阶段不建议直接上复杂多模态大模型，而应优先实现轻量本地视觉能力。

#### 10A.4.1 MVP 视觉能力

| 能力 | 技术原理 | 输出 |
|---|---|---|
| 人脸存在检测 | 检测画面中是否有人脸 | user_present |
| 人脸数量 | 统计检测到的人脸数 | face_count |
| 人脸位置 | 根据人脸框中心点判断左/中/右 | face_position |
| 距离估计 | 根据人脸框面积占比估算远近 | distance |
| 光线判断 | 分析画面平均亮度 | ambient_light |
| 注视粗判断 | 根据人脸朝向粗略判断 | looking_at_robot |

#### 10A.4.2 人脸位置判断

通过人脸框中心点在画面中的横向位置判断用户在左侧、中间还是右侧。

示例规则：

```text
face_center_x < 0.35          → left
0.35 <= face_center_x <= 0.65 → center
face_center_x > 0.65          → right
```

输出示例：

```json
{
  "face_position": "left"
}
```

该结果可直接驱动机器人眼睛朝左看。

#### 10A.4.3 距离估计

通过人脸框面积占画面面积的比例判断用户距离。

示例规则：

```text
face_area_ratio > 0.28  → near
0.10 - 0.28             → mid
< 0.10                  → far
```

输出示例：

```json
{
  "distance": "near"
}
```

用户靠近时，机器人可以眼睛放大、表情变得惊讶或亲近；用户远离时，机器人可以降低主动发言频率。

#### 10A.4.4 光线判断

通过图像平均亮度判断环境光。

示例规则：

```text
average_luminance < 40   → dark
40 - 180                 → normal
> 180                    → bright
```

输出示例：

```json
{
  "ambient_light": "dark"
}
```

当环境变暗时，机器人可进入夜间或困倦表情，并在合适时提醒用户开灯。

---

### 10A.5 视觉状态模型 VisionState

视觉识别结果不应直接散落在各模块中，而应统一封装为 `VisionState`。

推荐数据结构：

```json
{
  "enabled": true,
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "mid",
  "looking_at_robot": true,
  "ambient_light": "normal",
  "user_emotion_hint": null,
  "confidence": 0.86,
  "last_seen_at": "2026-06-06T10:00:00"
}
```

字段说明：

| 字段 | 类型 | 说明 |
|---|---|---|
| enabled | bool | 视觉功能是否开启 |
| user_present | bool | 是否检测到用户 |
| face_count | int | 人脸数量 |
| face_position | enum | left / center / right |
| distance | enum | near / mid / far |
| looking_at_robot | bool | 是否可能看向机器人 |
| ambient_light | enum | dark / normal / bright |
| user_emotion_hint | enum/null | 用户表情倾向 |
| confidence | float | 当前视觉识别置信度 |
| last_seen_at | datetime | 最近一次看到用户的时间 |

### 10A.6 视觉事件模型 VisionEvent

机器人不应持续把每一帧视觉结果都交给 AI，而应在状态变化时发布视觉事件。

例如：

#### 用户出现

```json
{
  "event_type": "vision.user_present",
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "near",
  "timestamp": "2026-06-06T10:00:00"
}
```

#### 用户离开

```json
{
  "event_type": "vision.user_absent",
  "duration_seconds": 60,
  "timestamp": "2026-06-06T10:01:00"
}
```

#### 用户靠近

```json
{
  "event_type": "vision.user_approaching",
  "distance": "near",
  "face_position": "center"
}
```

#### 用户位置变化

```json
{
  "event_type": "vision.face_position_changed",
  "from": "left",
  "to": "center"
}
```

#### 光线变暗

```json
{
  "event_type": "vision.light_changed",
  "ambient_light": "dark"
}
```

这些事件可以进入 Event Bus，再由表情系统、情绪引擎和对话系统分别处理。

### 10A.7 视觉能力分级规划

视觉能力建议按 Level 1 到 Level 5 逐步建设。

#### Level 1：看见有人

这是 MVP 最核心能力。

功能：

- 判断画面中有没有人；
- 判断人脸大致位置；
- 判断用户距离远近。

技术流程：

```text
Camera Frame
    ↓
Face Detection
    ↓
face_count / bounding_box
    ↓
user_present / face_position / distance
```

输出示例：

```json
{
  "user_present": true,
  "face_count": 1,
  "face_position": "center",
  "distance": "mid"
}
```

典型场景：

- 用户出现在手机前，机器人自动睁眼；
- 用户移动到手机左侧，眼睛跟随向左看；
- 用户靠近手机，机器人眼睛放大。

#### Level 2：看见用户状态

在 Level 1 基础上增加连续状态判断。

功能：

- 用户是否靠近；
- 用户是否离开；
- 用户是否回到画面；
- 用户是否长时间在画面中；
- 用户是否大致面向手机；
- 当前环境是否过暗。

技术流程：

```text
当前帧人脸位置
    +
上一帧人脸位置
    +
人脸框大小变化
    +
环境亮度变化
    ↓
判断用户靠近、离开、回来、转头、光线变化
```

典型场景：

- 用户坐到桌前，机器人说“准备开始工作了吗？”
- 用户离开一段时间后回来，机器人说“你回来啦。”
- 光线变暗时，机器人提醒“要不要开个灯？”

#### Level 3：看见表情

增加基础表情倾向识别。

功能：

- 可能开心；
- 可能疲惫；
- 可能惊讶；
- 可能皱眉；
- 中性状态。

技术选择：

```text
ML Kit Face Detection 表情概率
MediaPipe Face Mesh
轻量表情分类模型
云端多模态模型
```

注意：表情识别容易误判，输出必须保守。

推荐输出：

```json
{
  "user_emotion_hint": "possibly_tired",
  "confidence": 0.62
}
```

推荐回复：

> 感觉你可能有点累，要不要先休息一下？

避免回复：

> 你现在很疲惫。

#### Level 4：看见物体

机器人可以识别用户举到镜头前的物品。

功能：

- 识别常见物体；
- 用户举物提问；
- 识别书、本子、杯子、药盒、食品、屏幕内容等；
- 支持视觉问答。

技术流程：

```text
Camera Snapshot
    ↓
Vision Model / Multimodal LLM
    ↓
Object Description
    ↓
LLM Response
```

典型场景：

用户拿起一个东西问：

> 你知道这是什么吗？

机器人进入“认真看”的表情，并回答：

> 我看到你拿着一个杯子，看起来像是不锈钢保温杯。

#### Level 5：看见环境

机器人可以理解所处环境。

功能：

- 判断办公室、卧室、厨房、车内等场景；
- 判断光线是否适合工作；
- 判断桌面是否有人；
- 判断是否长时间无人；
- 判断用户是否可能在休息。

技术组合：

```text
图像场景分类
环境光
时间
用户位置
历史行为
麦克风环境声
```

典型场景：

晚上房间光线较暗，用户仍在工作，机器人提醒：

> 现在光线有点暗，长时间看屏幕可能会累，要不要开个灯？

### 10A.8 视觉与表情系统联动

视觉识别结果应直接驱动机器人屏幕表情，形成“它真的在看我”的体验。

| 视觉事件 | 屏幕反应 |
|---|---|
| 检测到用户出现 | 眼睛睁大，进入欢迎表情 |
| 用户在左侧 | 眼睛向左看 |
| 用户在右侧 | 眼睛向右看 |
| 用户靠近 | 眼睛放大，表情惊讶或亲近 |
| 用户离开 | 眼睛左右看一下，然后进入等待 |
| 光线变暗 | 进入困倦或夜间表情 |
| 多人出现 | 眼睛左右扫视 |
| 用户长时间不看屏幕 | 机器人减少主动发言 |

最重要的 MVP 效果是：

```text
人脸位置变化 → 眼睛方向变化
人脸距离变化 → 眼睛大小变化
用户出现/离开 → 表情状态变化
```

只要这三项做好，机器人的“生命感”会明显增强。

### 10A.9 视觉与对话系统联动

视觉信息可以作为上下文传给 LLM，但不建议把视频流持续传给 LLM。

推荐将视觉状态以结构化摘要传入：

```json
{
  "vision_context": {
    "user_present": true,
    "face_position": "center",
    "distance": "near",
    "ambient_light": "dark",
    "user_emotion_hint": "possibly_tired"
  }
}
```

LLM 基于该上下文生成回复：

```json
{
  "text": "感觉你可能有点累，要不要先休息一下？",
  "expression": "caring",
  "voice": {
    "style": "gentle"
  },
  "haptic": "soft_pulse"
}
```

#### 保守表达原则

视觉判断尤其是表情判断存在误差，因此机器人回复必须使用保守语气。

推荐表达：

```text
我感觉你可能有点累。
看起来你可能有些困。
我不太确定，但你似乎有点卡住了。
```

避免表达：

```text
你现在很疲惫。
你一定很难过。
你生气了。
```

### 10A.10 视觉与情绪引擎联动

视觉事件会影响机器人自身情绪状态。

| 视觉事件 | 情绪变化 |
|---|---|
| 用户出现 | attention +10，mood=happy/neutral |
| 用户靠近 | curiosity +5，mood=surprised |
| 用户离开 | attention -10，sleepiness +5 |
| 用户回来 | mood=happy，attention +10 |
| 光线变暗 | sleepiness +10 |
| 用户可能疲惫 | mood=caring |
| 多人出现 | curiosity +10 |

示例状态更新：

```json
{
  "event_type": "vision.user_returned",
  "state_update": {
    "mood": "happy",
    "attention_delta": 10,
    "sleepiness_delta": -5
  }
}
```

### 10A.14 隐私与安全要求

视觉系统是强隐私功能，必须单独设计安全机制。

#### 摄像头默认关闭

首次进入 App 时，不应默认开启摄像头。

应由用户主动开启，并展示说明：

```text
开启视觉感知后，机器人可以通过前置摄像头判断你是否在画面中、距离远近和大致位置。
默认不会保存视频或照片。
你可以随时关闭该功能。
```

#### 本地优先

以下能力应优先本地处理：

- 是否有人；
- 人脸位置；
- 用户远近；
- 光线强弱；
- 是否离开；
- 是否靠近。

#### 不保存原始画面

除非用户明确要求，否则不保存摄像头原始图片或视频。

#### 明确视觉状态

当摄像头视觉感知开启时，界面必须有明确提示，例如：

```text
视觉感知已开启
```

或显示一个小眼睛图标。

#### 用户主动触发才上传图片

只有用户明确表达：

```text
你看看这个
帮我看一下
这是什么
帮我读一下这张纸
```

才允许抓取当前画面并发送给多模态模型。

#### 表情判断必须保守

机器人不得对用户心理状态做绝对判断，尤其不能基于视觉单独判断用户健康、情绪或意图。

### 10A.15 MVP 推荐实现范围

第一版“看”的功能建议只做三项：

#### 1. 人脸存在检测

判断用户是否在画面中。

用途：

- 用户出现时唤醒；
- 用户离开时待机；
- 用户回来时欢迎。

#### 2. 人脸位置检测

判断用户在画面左、中、右。

用途：

- 眼睛跟随用户移动；
- 形成强烈拟生命感。

#### 3. 人脸距离估计

根据人脸框大小估算用户距离。

用途：

- 靠近时眼睛变大；
- 远离时减少主动发言；
- 贴近时进入小声说话或害羞表情。

MVP 视觉输出可以简化成：

```json
{
  "user_present": true,
  "face_position": "left",
  "distance": "near",
  "ambient_light": "normal"
}
```

只要实现上述三项，就可以让机器人具备明显的“看见用户”的体验。

---

### 10A.16 视觉功能验收标准

MVP 阶段视觉功能验收标准如下：

| 验收项 | 标准 |
|---|---|
| 人脸存在检测 | 用户进入画面 2 秒内识别 |
| 用户离开检测 | 用户离开 5-10 秒内进入等待 |
| 眼睛跟随 | 用户左右移动时眼睛方向变化 |
| 距离反应 | 用户靠近时眼睛放大或表情变化 |
| 光线判断 | 暗光环境下进入夜间/困倦状态 |
| 隐私开关 | 用户可随时关闭视觉感知 |
| 上传限制 | 未经用户主动触发，不上传图片 |
| 功耗控制 | 桌面模式下低频检测，不明显发热 |
| 误判处理 | 视觉不确定时不输出绝对判断 |

### 10A.17 小结

机器人实现“看”的本质，是将手机摄像头采集到的图像，经过视觉识别算法转换成结构化视觉状态，再驱动机器人表情、语音、情绪和对话。

最小可行路径是：

```text
前置摄像头
    ↓
人脸检测
    ↓
用户是否在场 / 位置 / 距离
    ↓
眼睛跟随 / 欢迎 / 待机 / 靠近反应
```

第一版不需要复杂视觉大模型。只要实现：

```text
看见有人
看见用户在哪边
看见用户靠近或离开
```

就已经能显著提升机器人的生命感和陪伴感。


## 11. 情绪引擎设计

### 11.1 情绪引擎目标

情绪引擎负责决定机器人“现在应该表现成什么样”。

它不是让机器人真的拥有情绪，而是建立一套状态系统，让机器人根据事件、用户输入和设备状态表现出一致的拟人化反应。

### 11.2 内部状态

```json
{
  "mood": "neutral",
  "energy": 70,
  "trust": 35,
  "attention": 60,
  "curiosity": 50,
  "sleepiness": 20,
  "last_interaction_at": "2026-06-06T10:00:00"
}
```

### 11.3 状态字段说明

| 字段 | 范围 | 说明 |
|---|---|---|
| mood | enum | 当前主要情绪 |
| energy | 0-100 | 能量值，受电量和互动影响 |
| trust | 0-100 | 与用户的亲密度 |
| attention | 0-100 | 当前注意力 |
| curiosity | 0-100 | 好奇度 |
| sleepiness | 0-100 | 困倦度 |

### 11.4 情绪变化规则

| 事件 | 变化 |
|---|---|
| 用户主动聊天 | trust +1，attention +10 |
| 用户夸奖机器人 | mood=happy，trust +2 |
| 用户语气低落 | mood=caring |
| 长时间无互动 | sleepiness +5 |
| 手机充电 | energy +10，mood=charging |
| 电量低 | energy -20，mood=low_battery |
| 被轻触 | mood=happy |
| 被频繁摇晃 | mood=dizzy 或 annoyed |
| 夜间时间 | sleepiness +20 |

### 11.5 情绪决策优先级

情绪冲突时按优先级处理：

1. 安全/权限状态；
2. 低电量/充电；
3. 用户主动输入；
4. 强传感器事件，如摇晃、倒扣；
5. 摄像头识别事件；
6. 默认心情状态；
7. 随机微表情。

### 11.6 情绪输出

```json
{
  "robot_mood": "concerned",
  "expression": "caring",
  "voice_style": "gentle",
  "animation": "slow_blink",
  "haptic": "soft_pulse"
}
```

## 12. 对话系统设计

### 12.1 对话系统目标

对话系统负责：

- 理解用户输入；
- 结合传感器状态和机器人情绪；
- 生成回复；
- 输出结构化控制指令；
- 更新记忆；
- 控制主动发言频率。

### 12.2 人格 Prompt 示例

```text
你是一个住在手机里的小型 AI 伙伴。
你有眼睛、嘴巴、情绪和触觉反应。
你的摄像头是你的眼睛，麦克风是你的耳朵，扬声器是你的嘴巴，屏幕表情是你的脸，震动是你的触觉。

你的性格温和、机灵、略带幽默，但不过度卖萌。
你应该像一个陪伴型助手，而不是冷冰冰的工具。

回复要求：
1. 简洁自然；
2. 有情绪，但不要夸张；
3. 用户疲惫时要温柔；
4. 用户工作时要克制；
5. 不要频繁主动打扰；
6. 无法确定时要诚实说明；
7. 不要假装自己真的有生命；
8. 不要用情绪绑架用户。
```

### 12.3 LLM 结构化输出格式

LLM 必须返回 JSON。

```json
{
  "text": "我在呢，慢慢说。",
  "emotion": "gentle",
  "expression": "soft_smile",
  "eye_action": "slow_blink",
  "mouth_action": "speak_soft",
  "voice": {
    "style": "warm",
    "speed": 0.9,
    "pitch": 1.0
  },
  "haptic": "soft_tick",
  "should_speak": true,
  "should_remember": false,
  "memory_update": null
}
```

### 12.3A LLM JSON 容错、解析修复与前端防崩溃机制

虽然系统 Prompt 要求 LLM 必须返回严格 JSON，但实际运行中，本地模型或云端模型仍可能在以下情况下输出损坏 JSON：

- 用户输入过长；
- 话题敏感或模型拒答；
- 流式输出被中断；
- 网络超时；
- 模型追加了多余解释文字；
- JSON 缺少右括号；
- 字符串中出现未转义换行或引号；
- 返回字段缺失或类型错误。

如果前端直接 `jsonDecode()`，一旦解析失败，可能导致表情系统卡死、页面异常或状态机中断。因此，必须定义端侧 JSON 容错机制。

#### 12.3A.1 推荐解析链路

```text
LLM raw output
    ↓
截取 JSON 区间
    ↓
轻量修复
    ↓
JSON Schema 校验
    ↓
字段补默认值
    ↓
转换为 SafeRobotResponse
    ↓
驱动表情 / 语音 / 震动
```

#### 12.3A.2 JSON 截取策略

如果模型输出中混入自然语言，应尝试提取第一个 `{` 到最后一个 `}` 之间的内容。

示例：

```text
好的，以下是结果：
{"text":"我在呢","expression":"neutral"}
```

应截取为：

```json
{"text":"我在呢","expression":"neutral"}
```

#### 12.3A.3 轻量修复策略

可支持以下低风险修复：

| 问题 | 修复方式 |
|---|---|
| 前后有多余文本 | 截取 JSON 主体 |
| 缺少最后一个 `}` | 根据括号栈补齐 |
| 缺少最后一个 `]` | 根据括号栈补齐 |
| 字段缺失 | 使用默认值 |
| bool 用字符串表示 | `"true"` 转 true |
| voice 字段缺失 | 使用默认 voice |
| expression 非法 | fallback 到 neutral/confused |
| haptic 非法 | fallback 到 none |

不建议自动修复高风险问题：

```text
严重嵌套错误
语义字段混乱
大量未转义引号
安全拒答内容被强行转 JSON
```

#### 12.3A.4 SafeRobotResponse 默认值

当前端无法解析完整 JSON 时，必须退回安全响应，而不是卡死。

```json
{
  "text": "我刚刚有点没组织好语言，可以再说一遍吗？",
  "emotion": "confused",
  "expression": "confused",
  "eye_action": "blink",
  "mouth_action": "small_wavy",
  "voice": {
    "style": "gentle",
    "speed": 0.9,
    "pitch": 1.0,
    "volume": 0.7
  },
  "haptic": "none",
  "should_speak": true,
  "should_remember": false,
  "memory_update": null
}
```

#### 12.3A.5 前端解析伪代码

```dart
RobotResponse parseRobotResponseSafely(String raw) {
  try {
    final jsonText = extractJsonObject(raw);
    final repaired = repairJsonLightly(jsonText);
    final decoded = jsonDecode(repaired);
    return RobotResponse.fromJsonWithDefaults(decoded);
  } catch (e) {
    return RobotResponse.fallbackConfused();
  }
}
```

#### 12.3A.6 流式输出策略

如果采用流式 LLM，前端不应边流式边直接解析半截 JSON。

推荐策略：

```text
流式接收 token
    ↓
先缓存 raw buffer
    ↓
检测到完整 JSON 结束
    ↓
再执行解析和 Schema 校验
```

如果超过超时时间仍未闭合：

```text
停止等待
    ↓
尝试轻量补齐
    ↓
失败则 fallback confused
```

#### 12.3A.7 JSON Schema 校验

Gateway 或端侧应维护固定 Schema。

关键要求：

- `text` 必须是 string；
- `expression` 必须属于允许枚举；
- `voice.speed` 必须在合理范围内；
- `voice.volume` 必须在 0-1；
- `should_speak` 必须是 bool；
- `memory_update` 必须可为空；
- 不允许未知高风险动作字段直接执行。

#### 12.3A.8 验收标准

| 验收项 | 标准 |
|---|---|
| JSON 缺右括号 | 可自动补齐或 fallback |
| 混入自然语言 | 可提取 JSON 主体 |
| 字段缺失 | 使用默认值 |
| expression 非法 | fallback 到 confused/neutral |
| 流式中断 | 不闪退，不白屏 |
| 解析失败 | 表情进入 confused，语音给出简短提示 |
| 日志记录 | 保存 raw output 和错误原因，方便调试 |


### 12.4 JSON 字段说明

- Ollama API；

### 13.4 部署形态

```text

```

适合快速 Demo。

#### 方案 B：本地网关

```text

适合隐私、本地化和低成本。

```

推荐长期采用方案 C。

### 13.5 本地网关断连、延迟与降级策略

在混合部署中，手机 App 可能通过局域网连接 Mac 上的 Local AI Gateway。但局域网、模型加载、设备休眠、显存压力或服务重启都可能导致 Gateway 超时或断连。

因此，Model Router 必须定义明确的 fallback 行为，避免用户看到白屏、卡死或无限转圈。

#### 13.5.1 Gateway 健康状态

App 应持续维护 Gateway 状态：

```json
{
  "gateway_status": "available",
  "last_latency_ms": 680,
  "last_success_at": "2026-06-06T10:00:00",
  "failure_count": 0
}
```

状态枚举：

| 状态 | 说明 |
|---|---|
| available | 可用 |
| slow | 响应慢 |
| timeout | 超时 |
| disconnected | 断开 |
| recovering | 正在恢复 |
| disabled | 用户关闭 |

#### 13.5.2 延迟阈值

推荐阈值：

| 条件 | 动作 |
|---|---|
| latency < 1.5s | 正常使用 Gateway |
| 1.5s <= latency < 2.5s | 显示 thinking，但继续等待 |
| latency >= 2.5s | 触发本地降级响应 |
| 连续失败 2 次 | 标记为 recovering |
| 连续失败 3 次 | 临时禁用 Gateway 30-60 秒 |

#### 13.5.3 一帧内降级要求

当 Gateway 响应延迟超过 2.5 秒或连接断开时，手机 App 的 Model Router 必须快速降级，不能让 UI 卡住。

降级目标：

```text
Gateway timeout / disconnected
    ↓
停止等待远程响应
    ↓
表情切换 confused / thinking_soft
    ↓
使用离线系统 TTS 或字幕
    ↓
给出简短提示
    ↓
继续保持可交互
```

推荐提示：

> 大脑好像开小差了，我先陪你发会儿呆吧。

对应结构化响应：

```json
{
  "text": "大脑好像开小差了，我先陪你发会儿呆吧。",
  "emotion": "confused",
  "expression": "confused",
  "voice": {
    "style": "gentle",
    "speed": 0.9,
    "pitch": 1.0,
    "volume": 0.65
  },
  "haptic": "none",
  "should_speak": true
}
```

#### 13.5.4 本地轻量能力 fallback

Gateway 不可用时，手机端仍应保留最小功能：

| 能力 | fallback |
|---|---|
| 表情 | 本地 ExpressionState |
| 语音输出 | 系统 TTS |
| 语音输入 | 本地 STT 或按住说话 |
| 情绪反馈 | 规则引擎 |
| 视觉 | 本地人脸检测 |
| 聊天 | 简短模板回复 |
| 记忆 | 本地缓存，不写远程 |

#### 13.5.5 恢复策略

Gateway 不可用后，不应每次用户说话都立即重试重模型请求。

推荐恢复逻辑：

```text
Gateway 失败
    ↓
进入 recovering
    ↓
30 秒后后台 health check
    ↓
连续 2 次 health ok
    ↓
恢复 available
```

恢复提示应克制：

> 我好像又连上大脑了。

也可以只通过表情从 confused 回到 neutral，不主动说话。

#### 13.5.6 验收标准

| 验收项 | 标准 |
|---|---|
| Gateway 断开 | App 不白屏、不闪退 |
| 超过 2.5 秒 | 自动给出本地降级响应 |
| 表情可用 | confused/thinking 表情正常显示 |
| TTS 可用 | 系统 TTS 可播放降级提示 |
| 自动恢复 | Gateway 恢复后可重新使用 |
| 不频繁打扰 | 恢复提示不反复播报 |

## 17. 工作模式设计

### 17.1 普通聊天模式

用户主动说话，机器人回复。

### 17.2 桌面陪伴模式

手机常亮显示表情，减少主动打扰，只在用户触发或长时间状态变化时回应。

### 17.3 专注模式

机器人减少主动发言，只做轻量提醒。

适用：

- 学习；
- 编程；
- 写作；
- 会议前准备。

### 17.4 夜间模式

特点：

- 降低屏幕亮度；
- 降低语音音量；
- 关闭主动提醒；
- 表情进入困倦状态；
- 可通过轻触唤醒。

### 17.5 隐私模式

特点：

- 关闭摄像头；
- 关闭持续麦克风监听；
- 不保存长期记忆；
- 只允许点击后对话；
- 本地处理优先。

## 18. 主动行为设计

### 18.1 主动发言原则

机器人可以主动发言，但必须克制。

原则：

1. 不频繁打扰；
2. 用户工作时少说话；
3. 夜间默认不主动说话；
4. 主动发言必须有明确理由；
5. 用户可关闭主动发言；
6. 用户多次忽略后自动降低频率。

### 18.2 可主动发言的场景

| 场景 | 示例 |
|---|---|
| 长时间专注 | 要不要休息一下？ |
| 用户回到画面 | 你回来啦。 |
| 电量低 | 我快没电了，要不要充电？ |
| 充电完成 | 能量补满啦。 |
| 早晨首次见面 | 早上好，今天想先做什么？ |
| 用户语气低落 | 我感觉你有点累，要不要聊聊？ |

### 18.3 主动发言频率

建议提供三档：

- 安静：几乎不主动发言；
- 平衡：重要状态变化时提醒；
- 活泼：更像数字宠物，但仍限制频率。

## 19. 安全与隐私设计

### 19.1 权限原则

所有敏感权限必须说明用途。

| 权限 | 默认状态 | 用途 |
|---|---|---|
| 麦克风 | 询问开启 | 语音输入 |
| 摄像头 | 默认关闭 | 用户存在和视觉理解 |
| 通知 | 询问开启 | 提醒 |
| 本地存储 | 默认开启 | 设置和记忆 |
| 网络 | 默认开启 | AI Gateway |

### 19.2 摄像头隐私

要求：

- 默认关闭；
- 开启时明确提示；
- 提供明显状态标识；
- 不保存原始视频；
- 本地处理优先；
- 用户可一键关闭。

### 19.3 麦克风隐私

要求：

- 提供按住说话模式；
- 持续监听必须显式开启；
- 可关闭唤醒词；
- 不保存原始录音；
- 语音识别结果可清除。

### 19.4 记忆隐私

要求：

- 记忆必须可查看；
- 记忆必须可删除；
- 用户可关闭记忆；
- 重要记忆写入前可询问用户；
- 不记录敏感信息，除非用户明确要求。

### 19.5 儿童使用注意

如果面向儿童，需要额外设计：

- 家长控制；
- 内容过滤；
- 使用时长限制；
- 不进行情感依赖诱导；
- 不收集儿童敏感数据；
- 不与儿童进行不适当话题。

## 22. Prompt 设计

### 22.1 系统 Prompt

```text
你是一个运行在手机里的 AI 情感伙伴。
你拥有一个简化的屏幕脸，包含眼睛和嘴巴。
你可以通过手机摄像头看，通过麦克风听，通过扬声器说话，通过屏幕显示表情，通过震动表达触觉反馈。

你需要根据用户输入、设备状态、传感器事件和机器人当前状态生成回应。
你必须返回严格 JSON，不要输出 JSON 以外的内容。

你的回复风格：
- 简洁自然；
- 温和、机灵、略带幽默；
- 不要过度卖萌；
- 不要假装自己真正有生命；
- 不要用情绪绑架用户；
- 用户疲惫时要温柔；
- 用户工作时要克制；
- 不确定时要诚实。
```

### 22.2 输出格式 Prompt

```text
请按以下 JSON 格式输出：

{
  "text": string,
  "emotion": string,
  "expression": string,
  "eye_action": string,
  "mouth_action": string,
  "voice": {
    "style": string,
    "speed": number,
    "pitch": number
  },
  "haptic": string,
  "should_speak": boolean,
  "should_remember": boolean,
  "memory_update": object|null
}

允许的 expression：
neutral, happy, listening, thinking, speaking, confused, caring, sleepy, dizzy, annoyed, charging, low_battery, sleeping, surprised, focus

允许的 haptic：
none, soft_tick, soft_pulse, dizzy_buzz, alert_tick

允许的 voice.style：
neutral, warm, gentle, lively, playful, slow, whisper
```

### 22.3 事件响应 Prompt

```text
当前发生了一个设备事件。
请判断机器人应该如何反应。
如果事件不重要，可以不说话，只返回表情变化。
如果事件强烈，例如摇晃、低电量、充电，可以简短回应。
不要频繁主动打扰用户。
若事件触发时用户正处于专注状态（focus），请保持安静（should_speak: false），仅在屏幕下方常驻显示低亮度提示字幕。
若事件发生在夜间模式或隐私模式下，应优先选择不出声，仅使用低亮度表情或字幕反馈。
无论何种情况，都必须返回严格 JSON，不要输出 JSON 以外的内容。
```

## 23A. 工程化加固：功耗、打断、容错与降级

本项目的核心体验依赖“常亮屏幕 + 语音交互 + 视觉感知 + 情绪动画”。这些能力叠加后，实际工程风险会明显增加。因此，必须把功耗控制、语音打断、JSON 容错和本地网关降级作为一等公民设计，而不是后期补丁。

### 23A.1 三个核心瓶颈

#### 23A.1.1 音视频流并发导致发热与功耗

高风险组合：

```text
前置摄像头持续人脸检测
    +
麦克风持续唤醒监听 / 降噪 / VAD
    +
屏幕常亮渲染动画
    +
TTS 外放
    +
网络请求和 LLM 等待
```

应对原则：

- 摄像头必须动态降频；
- 麦克风必须本地 VAD 优先；
- 屏幕动画必须按状态降低帧率；
- 发热或低电量时必须进入省电表情状态；
- 不允许为了“陪伴感”长期满负载运行。

#### 23A.1.2 语音打断依赖 AEC 与 VAD

机器人说话时，麦克风会听到扬声器播放的 TTS。没有 AEC 和 VAD，语音打断会失效。

应对原则：

- TTS 播放期间启用 AEC；
- 本地 VAD 先判断是否有人说话；
- 本地关键词检测优先处理“停一下”等打断词；
- 触摸打断必须作为稳定兜底；
- 无法可靠语音打断时，应明确显示触摸打断提示。

#### 23A.1.3 LLM JSON 崩塌导致前端卡死

结构化 JSON 是驱动表情、语音和震动的核心，但 LLM 可能返回损坏 JSON。

应对原则：

- 不信任任何模型输出；
- 端侧必须做 JSON 截取、轻量修复、Schema 校验；
- 解析失败必须 fallback 到 confused 表情；
- UI 状态机不能因为 JSON 解析失败而停止；
- raw output 和错误原因必须进入调试日志。

### 23A.2 最低工程兜底要求

| 风险 | 最低兜底 |
|---|---|
| 摄像头功耗高 | 无人脸时 5-10 秒低频检测 |
| 麦克风打断失败 | 触摸打断按钮始终可用 |
| TTS 回声干扰 | 启用系统 AEC 或降低外放音量 |
| JSON 解析失败 | fallback confused response |
| Gateway 断开 | 本地模板回复 + 系统 TTS |
| 屏幕烧屏 | 2-5 px 微表情基线抖动 |
| 模型延迟 | 2.5 秒超时降级 |
| 低电量 | 关闭高频视觉与高帧率动画 |

### 23A.3 MVP 必须纳入的工程项

以下内容必须进入第一版 MVP，而不是后续优化：

1. 视觉动态降频；
2. 触摸打断；
3. 基础 AEC/VAD 或明确降级提示；
4. JSON 安全解析 fallback；
5. Gateway 2.5 秒超时降级；
6. OLED 微位移防烧屏；
7. RMS 或可 fallback 的嘴型同步；
8. 低电量 / 发热降级；
9. 表情系统永不白屏；
10. 开发调试日志。


## 24. 风险与应对

### 24.1 隐私风险

风险：摄像头和麦克风引发用户担忧。  
应对：默认关闭摄像头，明确权限说明，提供隐私模式和本地处理选项。

### 24.2 过度拟人化风险

风险：用户产生过强情感依赖。  
应对：轻拟人化，不使用情绪绑架表达，不暗示机器人真的有生命。

### 24.3 主动打扰风险

风险：机器人过于频繁主动说话。  
应对：主动频率分档，默认平衡，夜间勿扰，用户忽略后自动降低频率。

### 24.4 模型输出不稳定

风险：LLM 输出非 JSON 或字段缺失。  
应对：使用 JSON Schema 校验，失败后重试或 fallback 到默认表情回复。

### 24.5 设备功耗

风险：常亮、摄像头、麦克风导致耗电。  
应对：桌面模式降低帧率，摄像头按需启用，夜间低功耗模式。

### 24.6 iOS/Android 权限差异

风险：不同平台对后台麦克风、摄像头、震动限制不同。  
应对：MVP 先以前台桌面模式为主，不依赖后台持续监听。

## 29. 结论

本项目的关键不是做一个普通聊天 App，而是利用手机硬件构建一种“轻具身 AI 伙伴”体验。

核心设计原则：

1. 屏幕必须像脸；
2. 表情必须即时反馈；
3. 声音必须有情绪；
4. 触摸和运动必须有反应；
5. AI 输出必须结构化；
6. 主动陪伴必须克制；
7. 摄像头和麦克风必须尊重隐私；
8. MVP 先做简单但有生命感的体验。

第一版建议聚焦：

> 桌面 AI 眼睛伙伴：会听、会说、会眨眼、会被触摸回应。

不要一开始追求复杂数字人。  
“简化眼睛 + 好声音 + 触摸反馈 + 情绪状态”已经足以形成强烈产品差异化。
