# 手机具身情感聊天机器人系统开发计划 V1.4

依据文档：`手机具身情感聊天机器人完整开发文档_V1.4.md`

目标：用分阶段方式完成“桌面 AI 眼睛伙伴”的 MVP，优先验证核心体验：会看起来活着、能听、能说、能用表情/震动/嘴型回应，并具备基础情绪和隐私控制。

## 1. 总体开发策略

### 1.1 MVP 核心闭环

第一版不追求复杂数字人、长期记忆图谱、多模态问答或完整本地模型平台。先跑通下面这个最小闭环：

```text
Flutter Face UI
  -> Touch / Sensor / Device Event
  -> ChatController
  -> AI Gateway / Cloud LLM
  -> Structured JSON
  -> ExpressionController
  -> TTS + Mouth Animation + Haptic
```

### 1.2 模块优先级

| 优先级 | 模块 | 说明 |
| --- | --- | --- |
| P0 | 表情系统 | 第一屏必须成立，机器人必须像“活着” |
| P0 | 语音输入/输出 | 支撑自然聊天 |
| P0 | 对话系统 | LLM 回复和结构化输出 |
| P0 | 情绪状态机 | 控制表情、语气、震动的一致性 |
| P0 | 触摸与运动感知 | 点击、长按、摇晃、倒扣等具身反应 |
| P0 | JSON 容错与降级 | 防止模型输出异常导致前端崩溃 |
| P0 | 权限与隐私基础 | 麦克风、摄像头、记忆都要可控 |
| P1 | 设置页、记忆页、桌面模式 | 完善可用性 |
| P2 | 摄像头视觉 | MVP 可预留，稳定后再增强 |
| P2 | 本地 AI Gateway | 后续形成本地大脑和混合路由 |

## 2. 推荐阶段计划

### 阶段 0：项目准备与技术定型

周期：2-3 天

目标：确定工程边界，建立前后端最小骨架。

主要任务：

- 确认 Flutter 为客户端主技术栈。
- 确认 AI Gateway 采用 FastAPI。
- 确认第一版优先支持前台桌面模式，不依赖后台持续监听。
- 创建客户端目录结构：face、voice、sensors、emotion、chat、settings、memory。
- 创建后端目录结构：DialogueManager、EmotionEngine、ModelRouter、MemoryManager。
- 定义统一响应结构 `SafeRobotResponse`。
- 定义 expression、voice.style、haptic、emotion 枚举。

交付物：

- Flutter 空项目可运行。
- FastAPI 空服务可运行。
- `/health` 接口可访问。
- 客户端能请求 Gateway 并拿到测试 JSON。

退出标准：

- 手机端主界面能启动。
- Gateway 能返回固定结构化响应。
- 前端能解析响应并显示基础状态。

### 阶段 1：表情壳子与具身事件

周期：1-2 周

目标：先让机器人“活起来”。

主要任务：

- 实现主界面 `FacePage`。
- 实现 `EyePainter` 和 `MouthPainter`。
- 实现基础表情：neutral、happy、listening、thinking、speaking、confused、caring、sleepy、dizzy、charging、low_battery。
- 实现眨眼、慢眨眼、眼神偏移、嘴巴开合。
- 实现 OLED 微位移防烧屏。
- 实现 `ExpressionState` 和 `ExpressionController`。
- 接入点击、双击、长按事件。
- 接入摇晃、拿起、倒扣、横竖屏识别。
- 接入电量和充电状态。
- 实现震动反馈：soft_tick、soft_pulse、dizzy_buzz、alert_tick。
- 实现表情调试页。

交付物：

- 可运行表情 Demo。
- 传感器事件 Demo。
- 表情调试页可以手动切换状态。

退出标准：

- 点击屏幕会眨眼。
- 长按进入 listening。
- 双击唤醒。
- 摇晃进入 dizzy。
- 倒扣进入 sleeping。
- 低电量和充电有对应表情。
- 表情系统在异常事件下不白屏、不冻结。

### 阶段 2：语音聊天与结构化 AI 输出

周期：1-2 周

目标：机器人能听懂、会说话，并且说话时嘴巴动。

主要任务：

- 接入麦克风权限。
- 接入 STT。
- 接入 TTS。
- 接入 Cloud LLM 或临时 AI Gateway。
- 实现 `ChatController`。
- 实现对话状态：idle、listening、thinking、speaking、interrupted、error。
- 实现 LLM 结构化 JSON 输出 Prompt。
- 实现 JSON 截取、轻量修复、Schema 校验、字段默认值。
- 实现 `SafeRobotResponse` fallback。
- 实现 TTS 播放状态回调。
- 实现 RMS 或简化嘴型同步。
- 实现触摸打断。
- 如平台支持，加入基础 AEC/VAD；不稳定时显示触摸打断入口。
- 实现 Gateway 2.5 秒超时降级。

交付物：

- 用户可以完成一次自然语音聊天。
- AI 返回内容可以控制表情、语音和震动。
- TTS 播放时嘴巴有同步动效。

退出标准：

- 语音输入可识别。
- AI 回复可转为结构化 JSON。
- TTS 可正常播放。
- 说话期间可以被触摸打断。
- JSON 损坏时不会崩溃，会进入 confused fallback。
- 模型或 Gateway 超时时有模板回复或安静降级。

### 阶段 3：情绪引擎与基础记忆

周期：1-2 周

目标：让机器人有连续状态，而不是每次都像重启。

主要任务：

- 设计并实现 `robot_state`：mood、energy、trust、attention、curiosity、sleepiness、last_interaction_at。
- 实现情绪优先级：安全/权限 > 电量/充电 > 用户输入 > 强传感器事件 > 视觉事件 > 默认心情 > 微表情。
- 实现事件驱动情绪变化规则。
- 实现设备事件到 emotion/expression/voice/haptic 的映射。
- 实现主动发言频率控制。
- 实现低打扰策略：专注模式、夜间模式、隐私模式。
- 实现简单记忆：用户名字、偏好、人格设置。
- 实现记忆开关和删除。

交付物：

- 机器人具备稳定人格和情绪表现。
- 用户偏好可以被保存和关闭。
- 情绪状态可以跨会话延续。

退出标准：

- 用户说“今天有点累”时进入 caring。
- 用户夸奖时进入 happy。
- 连续摇晃时从 dizzy 逐渐转为 annoyed。
- 夜间或专注模式下主动发言明显减少。
- 关闭记忆后不再写入长期记忆。

### 阶段 4：设置、页面与隐私控制

周期：3-5 天

目标：让 MVP 从 Demo 变成可长期试用的 App。

主要任务：

- 实现设置页。
- 实现人格选择或人格 Prompt 配置。
- 实现语音开关、摄像头开关、记忆开关、隐私模式。
- 实现桌面常亮模式。
- 实现低功耗模式。
- 实现日志页。
- 实现记忆页。
- 梳理权限说明与授权失败状态。

交付物：

- 用户可控制麦克风、摄像头、记忆、桌面模式。
- 开发者可通过日志定位问题。

退出标准：

- 未授权麦克风时不录音，并提示授权。
- 未授权摄像头时视觉功能不启动。
- 隐私模式会关闭摄像头和持续监听。
- 低电量时自动降低动画和视觉频率。

### 阶段 5：摄像头视觉感知

周期：2-3 周

目标：让机器人具备最基础的“看见用户”能力。

主要任务：

- 接入摄像头权限和预览/采集能力。
- 默认关闭摄像头，由用户显式开启。
- 实现人脸存在检测。
- 实现用户距离粗略判断。
- 实现用户靠近、离开、长时间不在场事件。
- 实现视觉事件与表情联动。
- 实现视觉事件与对话系统联动。
- 实现视觉动态降频：无人脸时低频检测，低电量/发热时关闭或降频。
- 预留多模态视觉问答接口。

交付物：

- 机器人知道用户是否在面前。
- 用户靠近时机器人能看向用户或轻微回应。

退出标准：

- 摄像头关闭时无采集。
- 用户出现/离开能触发事件。
- 视觉检测不会长期满负载运行。
- 夜间、隐私、低电量模式下视觉能力自动降级。

### 阶段 6：本地 AI Gateway 与混合路由

周期：2-4 周

目标：把手机变成前端终端，把 Mac 或本地机器作为“本地大脑”。

主要任务：

- 搭建 FastAPI Gateway。
- 实现 `/health`、`/chat`、`/event`、`/memory/query`、`/memory/delete`。
- 接入 Ollama / MLX 或其他本地模型。
- 实现云端 LLM fallback。
- 实现 ModelRouter：本地优先、云端兜底、超时降级。
- 实现 DialogueManager。
- 实现 EmotionEngine 服务端版本。
- 实现 MemoryManager 和 SQLite。
- 实现日志、错误追踪和请求耗时记录。
- 实现局域网连接配置。

交付物：

- 手机可以连接本地 Gateway。
- 普通对话可以通过本地模型完成。
- 本地失败时可切到云端或模板降级。

退出标准：

- `/chat` 返回严格结构化响应。
- `/event` 能处理设备事件。
- Gateway 断开时 App 不崩溃。
- 2.5 秒无响应时进入降级策略。
- 日志能看到模型、耗时、解析结果和 fallback 原因。

## 3. MVP 验收标准

第一版 MVP 完成时必须满足：

1. App 可稳定运行 30 分钟以上。
2. 主界面表情动画自然，帧率 30fps 以上。
3. 点击、长按、双击、摇晃、倒扣都有响应。
4. 可以完成一次完整语音对话。
5. AI 返回结构化 JSON。
6. JSON 异常时不会崩溃，有 confused fallback。
7. TTS 播放时嘴巴有同步动效。
8. 用户可以通过触摸打断说话。
9. 表情、语音、震动可以协同响应。
10. 低电量和充电状态有表情变化。
11. 用户可以关闭麦克风、摄像头、记忆。
12. 隐私模式可用。
13. Gateway 超时或断连时有降级策略。
14. 具备基础开发日志，方便调试。

## 6. 不建议第一版投入的内容

- 复杂 3D 数字人。
- 高精度唇形同步。
- 多人脸识别。
- 高级情绪识别。
- 完整长期记忆图谱。
- 儿童内容审核系统。
- 完整任务代理系统。
- App 内社交关系。
- 后台持续摄像头/麦克风监听。

## 7. 当前最推荐的第一步

先启动阶段 0 和阶段 1：

1. 建立 Flutter 项目与目录结构。
2. 建立 FacePage。
3. 用 CustomPainter 画出眼睛和嘴巴。
4. 实现表情状态机和几个核心表情。
5. 接入触摸、摇晃、倒扣、充电/低电量事件。
6. 做一个表情调试页。

这个阶段完成后，即使还没有 AI，也能判断产品的“具身感”和“陪伴感”是否成立。
